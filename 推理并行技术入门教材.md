## 1. 前言：为什么需要推理并行

### 1.1 问题场景

假设你有一个 **70B 参数**的大语言模型，以 FP16 精度推理。仅模型权重就需要：

```
70 × 10^9 × 2 bytes = 140 GB 显存
```

一张 H100（80GB 显存）装不下。这还只是模型本身——推理时你还需要：

| 显存消耗项 | 说明 | 典型大小 |
|-----------|------|----------|
| **模型权重** | 所有层的参数矩阵 | 70B × 2B = 140 GB (FP16) |
| **KV Cache** | 已生成 token 的 Key/Value 缓存 | ~2 × n_layers × n_heads × seq_len × d_head × 2B |
| **中间激活** | 前向计算中的临时张量 | 取决于 batch size 和序列长度 |
| **优化器状态** | （推理不需要） | 0 |

> 以 Llama-2-70B 为例处理 128K 上下文：仅 KV Cache 就需要 `2 × 80 × 64 × 131072 × 128 × 2 ≈ 320 GB`。

**结论：单卡根本跑不了。你需要多张 GPU 协同工作——这就是并行技术的用武之地。**

### 1.2 五种"分家"思路

```
┌──────────────────────────────────────────────────────────────────┐
│                     🏭 分布式并行全景图                             │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│   张量并行 (TP)    →  把权重矩阵切成多块，分 GPU 算                │
│   流水线并行 (PP)   →  把模型层分成多段，流水线处理                 │
│   数据并行 (DP)    →  把多个请求分给不同 GPU，各自独立推理          │
│   序列并行 (SP/CP) →  把输入序列切成多段，分 GPU 处理               │
│   专家并行 (EP)    →  把 MoE 的专家网络分给不同 GPU                │
│                                                                  │
│   本书聚焦 → TP / SP / CP / Ring Attention / Ulysses / PD分离     │
└──────────────────────────────────────────────────────────────────┘
```

## 2. 数学与概念基础

> 本章为入门读者准备。如果你熟悉 Transformer 和 Attention，可以直接跳到第3章。

### 2.1 Transformer 推理的基本流程

Transformer 推理的核心是 **逐 token 生成**：

```
输入: "今天天气"  →  [token_ids: 123, 456, 789, 101]
                         ↓
                  ┌─────────────┐
                  │  Embedding   │  → [B, S, D]  (batch=1, seq=4, hidden=4096)
                  └──────┬──────┘
                         ↓
                  ┌─────────────────────────────┐
                  │  Transformer Block × N_L    │  重复 80 层
                  │  ┌───────────────────────┐  │
                  │  │  LayerNorm            │  │
                  │  │  Attention (QKV投影)   │  │  ← 核心计算
                  │  │  Output Projection    │  │
                  │  │  LayerNorm            │  │
                  │  │  MLP (FFN)            │  │
                  │  └───────────────────────┘  │
                  └──────────────┬──────────────┘
                         ↓
                  ┌─────────────┐
                  │   LM Head    │  → [B, S, Vocab_Size]  → softmax → 下一个 token
                  └─────────────┘
```

### 2.2 Attention 机制详解

#### 2.2.1 单头注意力

对于输入 `X ∈ ℝ^(S×D)`，注意力计算如下：

```
Step 1: 线性投影
  Q = X @ W_Q    (S × D) @ (D × d_head) = (S × d_head)
  K = X @ W_K    (S × D) @ (D × d_head) = (S × d_head)
  V = X @ W_V    (S × D) @ (D × d_head) = (S × d_head)

Step 2: 计算注意力分数
  Scores = Q @ K^T / √d_head    (S × d_head) @ (d_head × S) = (S × S)

Step 3: Softmax（按行）
  Attention_Weights = softmax(Scores, dim=-1)    (S × S)

Step 4: 加权求和
  Output = Attention_Weights @ V    (S × S) @ (S × d_head) = (S × d_head)
```

#### 2.2.2 多头注意力（Multi-Head Attention, MHA）

```
输入: X ∈ ℝ^(S×D)  (D = n_heads × d_head)

投影: W_Q, W_K, W_V  ∈ ℝ^(D×D)

Q = X @ W_Q ∈ ℝ^(S×D)  →  reshape 为 (S, n_heads, d_head) → transpose → (n_heads, S, d_head)
K = X @ W_K ∈ ℝ^(S×D)  →  reshape 为 (S, n_heads, d_head) → transpose → (n_heads, S, d_head)
V = X @ W_V ∈ ℝ^(S×D)  →  reshape 为 (S, n_heads, d_head) → transpose → (n_heads, S, d_head)

每个 head 独立计算注意力：
  head_i = softmax(Q_i @ K_i^T / √d_head) @ V_i    (S, d_head)

所有 head 拼接并投影：
  MultiHead = concat([head_0, head_1, ..., head_{n-1}]) @ W_O    (S, D)
```

**QKV 张量的关键 Shape：**

| 阶段 | Q 的 Shape | K 的 Shape | V 的 Shape |
|------|-----------|-----------|-----------|
| 投影后 | `[B, S, D]` | `[B, S, D]` | `[B, S, D]` |
| Reshape | `[B, S, H, d]` | `[B, S, H, d]` | `[B, S, H, d]` |
| Transpose (MHA) | `[B, H, S, d]` | `[B, H, S, d]` | `[B, H, S, d]` |

> **符号约定**：B = Batch, S = Sequence Length, H = n_heads, d = d_head, D = H×d

#### 2.2.3 Grouped Query Attention (GQA) —— 现代模型的标配

Llama-2-70B 等模型使用 GQA：Q 有 64 个 head，但 K/V 只有 8 个 head。

```
Q: [B, H_q, S, d]    H_q = 64
K: [B, H_kv, S, d]    H_kv = 8
V: [B, H_kv, S, d]    H_kv = 8

每组 64/8 = 8 个 Q head 共享同一组 KV head
```

> **GQA 对并行策略有重要影响**：KV head 数量少限制了 Ulysses 等方案的并行度。

### 2.3 FlashAttention —— 所有并行方案的基础

FlashAttention 的核心思想：**不分块计算完整的 S×S 注意力矩阵，而是在 SRAM 中分块计算，避免将注意力矩阵写入 HBM。**

```
传统 Attention:    QK^T → S×S 矩阵写入 HBM → softmax 读回 → ×V
                    ↑ 这个 S×S 矩阵是显存杀手！

FlashAttention:   分块循环：
                   for block_i in Q_blocks:
                       for block_j in KV_blocks:
                           compute_softmax_incremental(block_i @ block_j^T, V_block_j)
                   ↑ 从未整体实例化 S×S 矩阵
```

**FlashAttention 的在线 softmax（Online Softmax）算法：**

```python
# 关键：不需要完整的 S×S 矩阵
# 维护 running max (m) 和 running sum (l)

m = -inf    # running max
l = 0       # running sum (normalizer)
O = 0       # output accumulator

for each KV_block:
    S_block = Q @ KV_block^T / sqrt(d)         # [B_r, B_c]
    m_new = max(m, S_block.max(dim=-1))        # 更新 max
    # 用数值稳定的方式更新
    l_new = exp(m - m_new) * l + (exp(S_block - m_new)).sum(dim=-1)
    P_block = softmax(S_block)                  # 在块内做
    O = diag(exp(m - m_new)) * O + P_block @ V_block
    m, l = m_new, l_new

O = diag(1/l) * O    # 最终归一化
```

> 💡 **重要**：FlashAttention 的在线 softmax 算法是所有分布式注意力（Ring Attention、Ulysses 的局部计算等）的数学基础。

### 2.4 Prefill 与 Decode 的本质区别

这是理解 PD 分离的关键：

```
┌─────────────────────────────────────────────────────────────────┐
│                    Prefill（预填充）                              │
├─────────────────────────────────────────────────────────────────┤
│  输入: 完整的 prompt（如 4096 个 token）                         │
│  处理: 一次性并行处理所有 token                                  │
│  Q 的 shape: [B, S_prompt, H, d]                                │
│  K 的 shape: [B, S_prompt, H_kv, d]                             │
│  QK^T: [B, H, S_prompt, S_prompt]                               │
│  计算特点: 矩阵乘法密集，compute-bound                            │
│  产出: 第一响应 token + 所有层的 KV Cache                         │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    Decode（逐 token 生成）                        │
├─────────────────────────────────────────────────────────────────┤
│  输入: 新生成的 1 个 token                                       │
│  处理: 用这个 token 的 Q 和全部历史 KV Cache 算注意力             │
│  Q 的 shape: [B, 1, H, d]                                       │
│  K 的 shape: [B, S_history + 1, H_kv, d]                        │
│  QK^T: [B, H, 1, S_history + 1]                                 │
│  计算特点: 显存带宽密集，memory-bound                             │
│  产出: 下一个 token + 追加后的 KV Cache                           │
└─────────────────────────────────────────────────────────────────┘
```

| 维度 | Prefill | Decode |
|------|---------|--------|
| **输入长度** | 长（128 ~ 128K+ tokens） | 短（1 token） |
| **瓶颈** | 计算（Compute-bound） | 显存带宽（Memory-bound） |
| **Attention pattern** | 全部 token 互相 attend | 1 个新 token attend 全部历史 |
| **并行机会** | 多（所有 token 可并行处理） | 少（逐 token） |
| **对通信敏感度** | 低（计算比通信大得多） | 高（通信延迟直接影响延迟） |

## 3. Tensor Parallelism（张量并行）—— Megatron 方案

### 3.1 一句话解释

> **把权重矩阵切成多块，每张 GPU 存一块、算一块，最后把结果拼起来。**

### 3.2 核心问题

模型权重太大，单卡放不下。比如一个 `4096×4096` 的权重矩阵（FP16 下占 32 MB），切成 4 份后每张 GPU 只存 `4096×1024`（8 MB）。

### 3.3 Megatron 的两种切分方式

Megatron-LM 为 Transformer 的每一层设计了两种 TP 切分方式，交替使用：

#### 3.3.1 列并行（Column Parallel）—— 用于 QKV 投影

**目标**：将 QKV 投影的权重 `W_QKV` 沿列切分。

```
原始权重: W_QKV ∈ ℝ^(D×D)   = ℝ^(4096×4096)
                               
列切分到 2 个 GPU:
  GPU 0: W_QKV[:, :D/2]  ∈ ℝ^(4096×2048)    ← 存前一半列
  GPU 1: W_QKV[:, D/2:]  ∈ ℝ^(4096×2048)    ← 存后一半列
```

**详细流程图（对照你的 Megatron TP 流程图）：**

```
输入: X ∈ ℝ^(B×S×D)  例: [1, 4096, 4096]

═══════════════════════════════════════════════════════════════
Step 1: f 和 g 的前向传播
═══════════════════════════════════════════════════════════════

X 是完整副本，每张 GPU 都有相同的 X（通过 broadcast 或 DP 获得）

  GPU 0                           GPU 1
  ┌──────────────────┐            ┌──────────────────┐
  │ X: [B,S,4096]    │            │ X: [B,S,4096]    │
  │ W[:,:2048]       │            │ W[:,2048:]       │
  │                  │            │                  │
  │ Y₀ = X @ W[:,:   │            │ Y₁ = X @ W[:,    │
  │      2048]       │            │      2048:]      │
  │    = [B,S,2048]  │            │    = [B,S,2048]  │
  └──────────────────┘            └──────────────────┘

  结果: Y = [Y₀ | Y₁] = [B, S, 4096]
  (逻辑上拼接，实际上分散在两个 GPU 上)

═══════════════════════════════════════════════════════════════
Step 2: 如果是 Attention 模块 —— QKV 的具体切分
═══════════════════════════════════════════════════════════════

QKV 权重 W_QKV ∈ ℝ^(4096 × 12288)   (4096 × 3×4096，3 = Q+K+V)

列切分到 4 个 GPU (TP=4):
  W_QKV 切为 [W_0 | W_1 | W_2 | W_3]，每块 (4096 × 3072)

  GPU 0 算了 1/4 的 Q、K、V → [B, S, 1024] 个 Q, [B, S, 1024] 个 K, [B, S, 1024] 个 V
  GPU 1 算了 1/4 的 Q、K、V → [B, S, 1024] 个 Q, [B, S, 1024] 个 K, [B, S, 1024] 个 V
  GPU 2 算了 1/4 的 Q、K、V → [B, S, 1024] 个 Q, [B, S, 1024] 个 K, [B, S, 1024] 个 V
  GPU 3 算了 1/4 的 Q、K、V → [B, S, 1024] 个 Q, [B, S, 1024] 个 K, [B, S, 1024] 个 V

  每个 GPU 的 Q/K/V 对应不同的 head 组：
  GPU 0 → heads [0, 0.25H]
  GPU 1 → heads [0.25H, 0.5H]
  GPU 2 → heads [0.5H, 0.75H]
  GPU 3 → heads [0.75H, H]

重排后每个 GPU 的 QKV shape：
  每个 GPU Q: [B, H/4, S, d]    ← 完整序列！只有部分 head
  每个 GPU K: [B, H/4, S, d]
  每个 GPU V: [B, H/4, S, d]

  每个 GPU 独立计算 attention（有限 head 的完整序列）：
  Attn_i = softmax(Q_i @ K_i^T / √d) @ V_i    [B, H/4, S, d]

═══════════════════════════════════════════════════════════════
Step 3: Output Projection —— 行并行
═══════════════════════════════════════════════════════════════

Attention 输出需要经过 W_O 投影回 D 维。

W_O ∈ ℝ^(D×D) = ℝ^(4096×4096)

行切分（Row Parallel）:
  GPU 0: W_O[:D/4, :]  ∈ ℝ^(1024×4096)
  GPU 1: W_O[D/4:D/2, :] ∈ ℝ^(1024×4096)
  ...

  每个 GPU 的 Attn_i ∈ [B, H/4, S, d]
  先 reshape 为 [B, S, D/4]   (其中 D/4 = H/4 × d)
  
  GPU i: Y_i = Attn_i_reshaped @ W_O_i    [B, S, D]  ← 但这是部分和的贡献

  然后 all-reduce 求和:
  Y = Y_0 + Y_1 + Y_2 + Y_3  → [B, S, D]

═══════════════════════════════════════════════════════════════
为什么列并行的输出不需要 all-reduce，而行并行需要？
═══════════════════════════════════════════════════════════════

列并行: Y = X @ [W₀|W₁] = [X@W₀ | X@W₁] → 天然就是拼接后的完整结果
         每张 GPU 拿着完整结果的一部分 → f (identity，不通信)
         
行并行: Y = [A₀|A₁] @ [W₀; W₁] = A₀@W₀ + A₁@W₁ → 需要求和才是完整结果
         每张 GPU 拿着完整结果的"部分贡献" → g (all-reduce 求和)
```

### 3.5 TP 的关键数学公式

设 P 为 TP 并行度：

**列并行：**
```
Y = X @ W    (完整: [B,S,D] @ [D,D] = [B,S,D])

列切分后:
  GPU_i: Y_i = X @ W_i    where W_i ∈ ℝ^(D × D/P)
  Y = concat([Y_0, Y_1, ..., Y_{P-1}], dim=-1)
```

**行并行 + all-reduce：**
```
Y = A @ W    (完整: [B,S,D] @ [D,D] = [B,S,D])

行切分后:
  GPU_i: Y_contrib_i = A_i @ W_i    where A_i ∈ ℝ^(B×S×D/P), W_i ∈ ℝ^(D/P × D)
  Y = sum(Y_contrib_i over i=0..P-1)   ← all-reduce
```

### 3.6 TP 的优势与局限

| 优势 | 局限 |
|------|------|
| 单卡显存 / P | 通信量大（每层都要 all-reduce） |
| 通信量与序列长度无关 | 需要高速互联（NVLink/NVSwitch） |
| 适配所有模型架构 | 跨节点通信太慢，只能单机用 |
| 适配所有序列长度 | **不解决长上下文问题**（每张 GPU 仍要处理完整序列） |

## 4. Sequence Parallelism（序列并行）—— Megatron-SP

### 4.1 与 TP 的区别

TP 切的是**权重**，每张 GPU 存部分模型但处理**完整序列**。序列一长，中间的激活值照样撑爆显存。

SP 切的是**序列**，每张 GPU 存完整模型（或完整 TP 组），但只处理**部分序列**。

### 4.2 Megatron-SP 的核心思想

```
原始输入: X ∈ ℝ^(B×S×D)   例: [1, 8192, 4096]

序列切分为 4 段:
  GPU 0: X[0:2048]  ∈ ℝ^(1×2048×4096)
  GPU 1: X[2048:4096] ∈ ℝ^(1×2048×4096)
  GPU 2: X[4096:6144] ∈ ℝ^(1×2048×4096)
  GPU 3: X[6144:8192] ∈ ℝ^(1×2048×4096)
```

### 4.3 Megatron-SP 的完整流程图

```
输入: X = [B, S, D] = [1, 8192, 4096]
序列切成 4 段

═══════════════════════════════════════════════════════════════
Attention 模块中的 SP
═══════════════════════════════════════════════════════════════

  每 GPU 本地序列: X_local = [B, S/P, D] = [1, 2048, 4096]

  ┌─────────────────────────────────────────────────────────────┐
  │  LayerNorm (序列相关！需要全局统计)                           │
  │  每 GPU 先算本地: μ_i = mean(X_local), σ²_i = var(X_local)   │
  │  all-reduce 得到全局 μ 和 σ²                                │
  │  本地 normalize: (X_local - μ) / √(σ² + ε)                  │
  │  输出: [B, S/P, D]                                          │
  └──────────────┬──────────────────────────────────────────────┘
                 ↓
  ┌─────────────────────────────────────────────────────────────┐
  │  QKV 投影 (如果用 TP，这里列并行；不用 TP 则完整)               │
  │  Q_local = X_local @ W_Q → [B, S/P, D]                      │
  │  K_local = X_local @ W_K → [B, S/P, D]                      │
  │  V_local = X_local @ W_V → [B, S/P, D]                      │
  └──────────────┬──────────────────────────────────────────────┘
                 ↓
  ┌─────────────────────────────────────────────────────────────┐
  │  ⚠️ 关键：Megatron-SP 的 Attention 仍算全局注意力！           │
  │  需要 all-gather Q/K/V 使每 GPU 得到完整序列                 │
  │                                                             │
  │  all-gather(Q_local) → Q_full [B, S, D]                    │
  │  all-gather(K_local) → K_full [B, S, D]                    │
  │  all-gather(V_local) → V_full [B, S, D]                    │
  │                                                             │
  │  然后正常算 Attention (完整序列)                              │
  └──────────────┬──────────────────────────────────────────────┘
                 ↓
  ┌─────────────────────────────────────────────────────────────┐
  │  Attention Output: [B, S, D] (完整序列，每 GPU 全量)         │
  │  取本地部分: [B, S/P, D]                                    │
  └──────────────┬──────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════════
MLP 模块中的 SP（Megatron-SP 的亮点）
═══════════════════════════════════════════════════════════════

  ┌─────────────────────────────────────────────────────────────┐
  │  LayerNorm (同样需要 all-reduce 得到全局均值和方差)           │
  │  → Z_local = [B, S/P, D]                                   │
  └──────────────┬──────────────────────────────────────────────┘
                 ↓
  ┌─────────────────────────────────────────────────────────────┐
  │  MLP 上投影 (W_up): [B, S/P, D] @ W_up → [B, S/P, D_ffn]   │
  │  ✅ 序列独立！不需要通信                                     │
  │                                                             │
  │  Activation (SiLU/GeLU): [B, S/P, D_ffn]                   │
  │  ✅ 序列独立！不需要通信                                     │
  │                                                             │
  │  MLP 下投影 (W_down): [B, S/P, D_ffn] @ W_down              │
  │  → [B, S/P, D]  ✅ 序列独立！不需要通信                      │
  └──────────────┬──────────────────────────────────────────────┘

  ⚡ Megatron-SP 的关键节省：MLP 的中间激活只占 S/P
  原来 MLP 中间: [B, S, D_ffn]     → 占用 S×D_ffn 空间
  SP 后 MLP 中间: [B, S/P, D_ffn]  → 占用只有 1/P
```

### 4.4 Megatron-SP 的通信模式总结

```
每层 Transformer Block 的通信:

  Attention:
    LayerNorm: all-reduce (为得到全局 μ, σ²)
    Q/K/V all-gather (为算全局注意力)
    Output: 取本地部分（不通信）

  MLP:
    LayerNorm: all-reduce (为得到全局 μ, σ²)
    其余全本地，无需通信！
```

### 4.5 SP 的局限

> Megatron-SP 只在 MLP 层真正节省了激活值显存。Attention 部分仍然需要 `all-gather` 出完整序列来算全局注意力——这本质上还是在算 **O(S²)** 的注意力。当序列超长时，这点不够。

## 5. Context Parallelism（上下文并行）—— DeepSpeed Ulysses

> ⭐ **本章为重点章节**，详细展示 QKV 张量的每一次 shape 变化。

### 5.1 核心思想

Ulysses 走出了一条和 Ring Attention 完全不同的路：

```
Ring Attention: "把 KV 传给需要它们的 GPU" → P2P 环状传递
Ulysses:        "用 all-to-all 重新排列 QKV，让每个 GPU 恰好有完整的序列
                 但只有部分 head → 各自本地算注意力 → 再 all-to-all 排回来"
```

### 5.2 为什么叫 Ulysses？

来自希腊神话《奥德赛》（Odyssey）。Ulysses 经历漫长漂泊回到家乡，对应数据在 GPU 之间"旅行"后回到原位的过程。

---

### 5.3 Ulysses 的 Prefill 详细流程图

```

═══════════════════════════════════════════════════════════════════════

配置:
  序列长度 S = 8192
  Head 数 H = 32
  Head 维度 d = 128
  隐藏维度 D = H × d = 4096
  上下文并行度 CP = 4

═══════════════════════════════════════════════════════════════════
阶段0: 初始状态 —— 按序列切分
═══════════════════════════════════════════════════════════════════

输入 X: [B, S, D] = [1, 8192, 4096]
序列切成 4 段:

  GPU 0: X₀ = X[:, 0:2048, :]      → [1, 2048, 4096]  (token 0~2047)
  GPU 1: X₁ = X[:, 2048:4096, :]   → [1, 2048, 4096]  (token 2048~4095)
  GPU 2: X₂ = X[:, 4096:6144, :]   → [1, 2048, 4096]  (token 4096~6143)
  GPU 3: X₃ = X[:, 6144:8192, :]   → [1, 2048, 4096]  (token 6144~8191)

═══════════════════════════════════════════════════════════════════
阶段1: QKV 投影 —— 每 GPU 本地计算
═══════════════════════════════════════════════════════════════════

每 GPU 用自己那截序列做 QKV 投影：

  GPU 0: Q₀ = X₀ @ W_Q → [1, 2048, 4096]
          K₀ = X₀ @ W_K → [1, 2048, 4096]

  GPU 1: Q₁ = X₁ @ W_Q → [1, 2048, 4096]
          K₁ = X₁ @ W_K → [1, 2048, 4096]
          V₁ = X₁ @ W_V → [1, 2048, 4096]

  GPU 2: Q₂ = X₂ @ W_Q → [1, 2048, 4096]
          K₂ = X₂ @ W_K → [1, 2048, 4096]
          V₂ = X₂ @ W_V → [1, 2048, 4096]

  GPU 3: Q₃ = X₃ @ W_Q → [1, 2048, 4096]
          K₃ = X₃ @ W_K → [1, 2048, 4096]
          V₃ = X₃ @ W_V → [1, 2048, 4096]

═══════════════════════════════════════════════════════════════════
阶段2: Reshape —— 从扁平化到多头视图
═══════════════════════════════════════════════════════════════════

每 GPU 把 [B, S_local, D] reshape 为 [B, S_local, H, d]:

  GPU 0: Q₀ = reshape([1, 2048, 4096]) → [1, 2048, 32, 128]
          K₀ = reshape([1, 2048, 4096]) → [1, 2048, 32, 128]
          V₀ = reshape([1, 2048, 4096]) → [1, 2048, 32, 128]

  GPU 1: Q₁ = [1, 2048, 32, 128]
  GPU 2: Q₂ = [1, 2048, 32, 128]
  GPU 3: Q₃ = [1, 2048, 32, 128]

═══════════════════════════════════════════════════════════════════
阶段3: ★ 第一次 All-to-All —— Ulysses 的核心！★
═══════════════════════════════════════════════════════════════════

目的: 把数据从 "按序列切分" 转换为 "按 head 切分"

输入每个 GPU: [B, S_local, H, d] = [1, 2048, 32, 128]
输出每个 GPU: [B, S_local, H/CP, d] = [1, 2048, 8, 128]
              ↑ 但来自不同 GPU 的不同序列段！

具体操作:
  把 H=32 个 head 均匀分给 CP=4 个 GPU
  GPU 0 负责 head 0~7
  GPU 1 负责 head 8~15
  GPU 2 负责 head 16~23
  GPU 3 负责 head 24~31

All-to-All 的详细数据流动（以 Q 为例）:

  ┌─────────────────────────────────────────────────────────────┐
  │ 发送前 (每个 GPU: [1, 2048, 32, 128])                        │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ GPU 0: Q[token0..2047, heads 0..31]                         │
  │         ├── heads 0..7  → 留在 GPU 0  (本地)                 │
  │         ├── heads 8..15 → 发给 GPU 1                        │
  │         ├── heads 16..23 → 发给 GPU 2                       │
  │         └── heads 24..31 → 发给 GPU 3                       │
  │                                                             │
  │ GPU 1: Q[token2048..4095, heads 0..31]                      │
  │         ├── heads 0..7  → 发给 GPU 0                        │
  │         ├── heads 8..15 → 留在 GPU 1  (本地)                 │
  │         ├── heads 16..23 → 发给 GPU 2                       │
  │         └── heads 24..31 → 发给 GPU 3                       │
  │                                                             │
  │ GPU 2, GPU 3: 同理 (省略)                                    │
  └─────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────┐
  │ 接收后 (每个 GPU: [1, 2048×4, 8, 128])                       │
  ├─────────────────────────────────────────────────────────────┤
  │        ↑ 来自所有 GPU 的不同序列段                            │
  │                                                             │
  │ GPU 0: Q[heads 0..7, token0..8191]                          │
  │        来自 GPU 0 的 token0..2047, heads 0..7                │
  │        来自 GPU 1 的 token2048..4095, heads 0..7             │
  │        来自 GPU 2 的 token4096..6143, heads 0..7             │
  │        来自 GPU 3 的 token6144..8191, heads 0..7             │
  │                                                             │
  │        → 拼接: [1, 2048×4, 8, 128] = [1, 8192, 8, 128]      │
  │        → 重排: [1, 8, 8192, 128]                            │
  │                                                             │
  │ GPU 1: Q[heads 8..15, token0..8191]                         │
  │        → [1, 8, 8192, 128]                                  │
  │                                                             │
  │ GPU 2: Q[heads 16..23, token0..8191]                        │
  │        → [1, 8, 8192, 128]                                  │
  │                                                             │
  │ GPU 3: Q[heads 24..31, token0..8191]                        │
  │        → [1, 8, 8192, 128]                                  │
  └─────────────────────────────────────────────────────────────┘

All-to-All 后的 QKV Shape 总结:

  GPU 0: Q=[1, 8, 8192, 128]  K=[1, 8, 8192, 128]  V=[1, 8, 8192, 128]
  GPU 1: Q=[1, 8, 8192, 128]  K=[1, 8, 8192, 128]  V=[1, 8, 8192, 128]
  GPU 2: Q=[1, 8, 8192, 128]  K=[1, 8, 8192, 128]  V=[1, 8, 8192, 128]
  GPU 3: Q=[1, 8, 8192, 128]  K=[1, 8, 8192, 128]  V=[1, 8, 8192, 128]

  ✅ 每 GPU 都有完整序列 (S=8192)，但只有 1/4 的 head (8 个)！
  ✅ 每个 GPU 可以本地算注意力，不需要任何通信！

═══════════════════════════════════════════════════════════════════
阶段4: 本地计算 Attention (完全独立，无通信！)
═══════════════════════════════════════════════════════════════════

每 GPU 独立计算:

  GPU 0:
    attn_scores = Q₀ @ K₀^T / √128    → [1, 8, 8192, 8192]
    attn_weights = softmax(attn_scores) → [1, 8, 8192, 8192]
    output = attn_weights @ V₀          → [1, 8, 8192, 128]

  GPU 1: (同样，各自算)  → [1, 8, 8192, 128]
  GPU 2: (同样，各自算)  → [1, 8, 8192, 128]
  GPU 3: (同样，各自算)  → [1, 8, 8192, 128]

  注意：每 GPU 的 head 组不同，互不重叠 → 合在一起就是完整的 multi-head attention！

═══════════════════════════════════════════════════════════════════
阶段5: ★ 第二次 All-to-All —— 把结果排回序列视图 ★
═══════════════════════════════════════════════════════════════════

目的: 从 "按 head 切分" 转回 "按序列切分"

输入:  每 GPU: [1, 8, 8192, 128]  (所有序列, 部分 head)
输出:  每 GPU: [1, 2048, 32, 128]  (部分序列, 所有 head)

操作: 第一次 all-to-all 的逆操作

  GPU 0 发送: [heads 0..7] 中属于 token0..2047 的部分 → 留在本地
              [heads 0..7] 中属于 token2048..4095 的部分 → 发给 GPU 1
              [heads 0..7] 中属于 token4096..6143 的部分 → 发给 GPU 2
              [heads 0..7] 中属于 token6144..8191 的部分 → 发给 GPU 3

  (同时从所有 GPU 接收它们算好的 head 组的对应序列块)

  GPU 0: O₀ = [1, 2048, 32, 128]  (token 0~2047, 所有 32 个 head)
  GPU 1: O₁ = [1, 2048, 32, 128]  (token 2048~4095, 所有 32 个 head)
  GPU 2: O₂ = [1, 2048, 32, 128]
  GPU 3: O₃ = [1, 2048, 32, 128]

═══════════════════════════════════════════════════════════════════
阶段6: Reshape + Output Projection
═══════════════════════════════════════════════════════════════════

  GPU 0: reshape([1, 2048, 32, 128]) → [1, 2048, 4096]
          O_proj = O₀ @ W_O → [1, 2048, 4096]  (本地完成)

  GPU 1/2/3: 同理

最终每 GPU 输出: [B, S/CP, D] = [1, 2048, 4096]
这就是对应自己那截序列的 Attention 输出。
```

### 5.4 Ulysses 的 Decode 流程

Decode 阶段与 Prefill 几乎一致，区别在于 Q 的长度：

```
═══════════════════════════════════════════════════════════════════
            Ulysses —— Decode 阶段
═══════════════════════════════════════════════════════════════════

配置:
  已缓存 KV Cache (全部历史 token)
  新生成 1 个 token (当前在 GPU 0 上)

阶段1: 新 token 的 QKV 投影
  产生新 token 的 GPU (GPU 0):
    q_new = [1, 1, 32, 128]   ← 只有 1 个 token
    k_new = [1, 1, 32, 128]
    v_new = [1, 1, 32, 128]

  其他 GPU: 没有新 token (或通过广播得到)

阶段2: 第一次 All-to-All (Q 的分布在序列维度上不均衡)
  因为只有 1 个新 token，Q 很小 → all-to-all 开销量极小
  q_new 按 head 切片分发到各 GPU

阶段3: 各 GPU 追加 k_new, v_new 到本地 KV Cache
  GPU i: KV_cache_i = concat(KV_cache_i, kv_new[:, heads_i, :, :])

阶段4: 本地算 Attention
  每 GPU: attn(q_new[:, heads_i, :, :], KV_cache_i) → 只 attend 1 个 query token

阶段5: 第二次 All-to-All 排回
  
  因为 O 很小 (只有 1 个 token)，通信量极小

💡 Ulysses 在 Decode 阶段的优势:
  - Decode 时 Q 只有 1 个 token → 两轮 all-to-all 通信量极小
  - 每轮 all-to-all 只传 head_size × d × 1 token → ≈ 4096 floats
  - 通信延迟是瓶颈，但带宽不是
```

### 5.5 Ulysses 的核心优势与限制

| 优势 | 限制 |
|------|------|
| Attention 计算完全本地化，无需通信 | **Head 数必须 ≥ CP 并行度** |
| 通信只有 2 轮 all-to-all（恒定轮数） | GQA 模型的 KV head 少，限制 CP 并行度 |
| 支持任意 Attention 变体（MHA/GQA/MQA/FlashAttention） | All-to-All 对网络拓扑敏感 |
| Decode 阶段通信量极小 | 计算与通信不能重叠（必须先通信完再算） |
| 代码实现简洁（相比 Ring Attention 的异步流水线） | 不适用 head 少的场景（如 MQA: head=1） |

### 5.6 Ulysses 通信量数学分析

```
设:
  B = batch size
  S = 序列总长度
  S_local = S / CP
  H = head 总数
  d = head 维度
  CP = 上下文并行度

第一轮 All-to-All:
  每个 GPU 发送: B × S_local × H × d × 3 (Q, K, V)
  分散到 CP 个 GPU 后，每 GPU 接收: B × S_local × H × d × 3

第二轮 All-to-All:
  每个 GPU 发送: B × S × H/CP × d (输出 O)
  分散到 CP 个 GPU 后，每 GPU 接收: B × S_local × H × d

总通信量/GPU:
  = B × S_local × H × d × (3 + 1)
  = B × S_local × D × 4    (因为 D = H × d)
  = 4 × B × S × D / CP
```

## 6. TP + Ulysses 组合并行

> ⭐ **本章为重点章节**。当模型大（需要 TP）且上下文长（需要 CP），就需要 TP + Ulysses 组合。

### 6.1 为什么需要组合？

```
场景: Llama-2-70B, 32 层, 128K 上下文, 8 张 H100

纯 TP (TP=8):
  ✅ 模型装得下 (140GB / 8 = 17.5GB / GPU)
  ❌ 128K 序列 → 注意力矩阵 128K² = 16G entries 撑爆显存

纯 Ulysses CP (CP=8):
  ✅ 每 GPU 处理 16K token 的注意力 → 注意力矩阵 16K² = 256M entries
  ❌ Llama-2-70B 的 KV head 只有 8 个！CP=8 时每 GPU 只分到 1 个 KV head
     → 但 Q head 有 64 个，每 GPU 分到 8 个 Q head
     → GQA 下这是可行的！但模型权重仍然太大

TP + Ulysses 组合:
  TP=2, CP=4 (共 8 GPU)
  ✅ TP=2 分担模型权重的 50%
  ✅ CP=4 分担序列的 75%
  ✅ 每 GPU 只处理 32K 的序列段
```

### 6.3 TP + Ulysses 的 Prefill 详细流程

```
═══════════════════════════════════════════════════════════════════
          TP + Ulysses 组合 —— Prefill 详细流程
═══════════════════════════════════════════════════════════════════

配置:
  TP=2, CP=4, 共 8 GPU
  序列 S=8192, H=32, d=128, D=4096
  S_local = S/CP = 2048

阶段0: 初始状态
  每个 CP 组内的 2 个 TP GPU 拿到相同的序列段
  CP Group 0 (GPU 0,1): X[:, 0:2048, :]
  CP Group 1 (GPU 4,5): X[:, 2048:4096, :]
  CP Group 2 (GPU 2,6): X[:, 4096:6144, :]
  CP Group 3 (GPU 3,7): X[:, 6144:8192, :]

═══════════════════════════════════════════════════════════════════
阶段1: TP 下的 QKV 投影 + CP 的第一次 All-to-All
═══════════════════════════════════════════════════════════════════

Step 1.1: TP 列并行计算 QKV (每 CP 组内)

  以 CP Group 0 (GPU 0,1) 为例:

  GPU 0 (TP rank 0):
    Q_part0 = X_local @ W_Q[:, :D/2] → [1, 2048, 2048]
    K_part0 = X_local @ W_K[:, :D/2] → [1, 2048, 2048]
    V_part0 = X_local @ W_V[:, :D/2] → [1, 2048, 2048]
    → reshape 为 [1, 2048, 16, 128]  (H/2 = 16 个 head)

  GPU 1 (TP rank 1):
    Q_part1 = X_local @ W_Q[:, D/2:] → [1, 2048, 2048]
    K_part1 = X_local @ W_K[:, D/2:] → [1, 2048, 2048]
    V_part1 = X_local @ W_V[:, D/2:] → [1, 2048, 2048]
    → reshape 为 [1, 2048, 16, 128]  (另外 16 个 head)

  ⚠️ TP 组内的两个 GPU 各自持有不同的 head 组，但共享同一段序列！

Step 1.2: 所有 CP 组同时做第一次 All-to-All (跨 CP, 相同 TP rank)

  把 "序列维度" 重排为 "head 维度":

  TP rank 0 的 4 个 GPU (GPU 0,4,2,6) 之间做 all-to-all:
    GPU 0: heads 0..15, token 0..2047
    GPU 4: heads 0..15, token 2048..4095
    GPU 2: heads 0..15, token 4096..6143
    GPU 6: heads 0..15, token 6144..8191

    交换后:
    每 GPU 分到 H/(TP×CP) = 32/(2×4) = 4 个 head
    但拥有完整序列 token 0..8191

  TP rank 1 的 4 个 GPU (GPU 1,5,3,7) 同理

All-to-All 后的完整 Shape 分布:

  GPU 0 (TP=0, CP=0): Q=[1, 4, 8192, 128], K=[1, 4, 8192, 128], V=[1, 4, 8192, 128]
  GPU 1 (TP=1, CP=0): Q=[1, 4, 8192, 128], K=[1, 4, 8192, 128], V=[1, 4, 8192, 128]
  GPU 4 (TP=0, CP=1): Q=[1, 4, 8192, 128], K=[1, 4, 8192, 128], V=[1, 4, 8192, 128]
  GPU 5 (TP=1, CP=1): Q=[1, 4, 8192, 128], K=[1, 4, 8192, 128], V=[1, 4, 8192, 128]
  GPU 2 (TP=0, CP=2): Q=[1, 4, 8192, 128], K=[1, 4, 8192, 128], V=[1, 4, 8192, 128]
  GPU 6 (TP=1, CP=2): Q=[1, 4, 8192, 128], K=[1, 4, 8192, 128], V=[1, 4, 8192, 128]
  GPU 3 (TP=0, CP=3): Q=[1, 4, 8192, 128], K=[1, 4, 8192, 128], V=[1, 4, 8192, 128]
  GPU 7 (TP=1, CP=3): Q=[1, 4, 8192, 128], K=[1, 4, 8192, 128], V=[1, 4, 8192, 128]

  ✅ 每个 GPU: 完整序列 (8192) + 4 个 head
  ✅ 每 GPU 可以本地独立计算自己的 4 个 head 的注意力

═══════════════════════════════════════════════════════════════════
阶段2: 本地计算 Attention
═══════════════════════════════════════════════════════════════════

  每 GPU 独立: attn_out = FlashAttention(Q, K, V)
  Shape: [1, 4, 8192, 128]

═══════════════════════════════════════════════════════════════════
阶段3: 第二次 All-to-All (排回序列分布)
═══════════════════════════════════════════════════════════════════

  把 [B, H_local, S, d] 转回 [B, S_local, H, d] 按序列分布

  GPU 0 (TP=0, CP=0): 输出 [1, 2048, 16, 128] ← 序列 0~2047, TP rank 0 的 head
  GPU 1 (TP=1, CP=0): 输出 [1, 2048, 16, 128] ← 序列 0~2047, TP rank 1 的 head
  ...

═══════════════════════════════════════════════════════════════════
阶段4: Output Projection (TP 行并行 + all-reduce)
═══════════════════════════════════════════════════════════════════

  TP 组内 (GPU 0 和 GPU 1):
    各自 reshape: [1, 2048, 16, 128] → [1, 2048, 2048]
    各自 W_O 投影 (行并行): o_i = attn_i @ W_O_i → [1, 2048, 4096]
    组内 all-reduce: o = o_0 + o_1 → [1, 2048, 4096]

最终输出:
  GPU 0,1: [1, 2048, 4096]  (序列段 0~2047)
  GPU 4,5: [1, 2048, 4096]  (序列段 2048~4095)
  GPU 2,6: [1, 2048, 4096]  (序列段 4096~6143)
  GPU 3,7: [1, 2048, 4096]  (序列段 6144~8191)
```

### 6.4 TP + Ulysses 的通信顺序总结

```
每层 Transformer 的通信:

┌─────────────────────────────────────────────────────────────────┐
│ 1. QKV 投影: TP 列并行 → f (identity), 不通信                    │
│ 2. 第一次 All-to-All: CP 组内通信 → 序列→head 重排               │
│ 3. Attention 计算: 完全本地，不通信                               │
│ 4. 第二次 All-to-All: CP 组内通信 → head→序列 重排               │
│ 5. W_O 投影: TP 行并行 → g (all-reduce), TP 组内通信             │
│ 6. MLP 块: TP 列并行(f) + TP 行并行(g, all-reduce)               │
└─────────────────────────────────────────────────────────────────┘

TP 通信: 只在同 CP 组内的 TP 对之间 (通常 NVLink 直连)
CP 通信: 只在同 TP rank 的 GPU 之间 (可能跨节点)
```

### 6.5 GQA 下 TP + Ulysses 的特殊考量

```
Llama-2-70B GQA 配置:
  Q heads: 64
  K/V heads: 8

如果 TP=2, CP=4:
  Q: 每 GPU 分到 64/(2×4) = 8 个 Q head
  K/V: 每 GPU 分到 8/(2×4) = 1 个 KV head

可以运行！但要确保：
  K/V head 总数 ≥ CP (此处 8 ≥ 4 ✅)
  
如果改为 CP=8:
  K/V: 8/8 = 1 个 KV head/GPU ✅ (刚好)
  但每 GPU 的 Q head: 64/(2×8) = 4, 仍然可行

极端情况 CP=16:
  K/V: 8/16 = 0.5 → ❌ 不够分！GQA 限制了 Ulysses 的扩展性
```

## 7. Ring Attention —— 环状注意力

### 7.1 核心思想

> **GPU 排成一个环，KV 块一圈圈地传。每传一圈，每个 GPU 就用新收到的 KV 算一步注意力——边传边算，计算和通信重叠。**

### 7.2 Ring Attention 的 Prefill 详细流程

```
═══════════════════════════════════════════════════════════════════
          Ring Attention —— Prefill 详细流程
═══════════════════════════════════════════════════════════════════

配置: GPU=4, 序列 S=8192, d=128
序列切成 4 段: 每 GPU 处理 2048 个 token

初始状态:
  GPU 0: Q₀, K₀, V₀ ∈ [1, 2048, 128]  (token 0~2047)
  GPU 1: Q₁, K₁, V₁ ∈ [1, 2048, 128]  (token 2048~4095)
  GPU 2: Q₂, K₂, V₂ ∈ [1, 2048, 128]  (token 4096~6143)
  GPU 3: Q₃, K₃, V₃ ∈ [1, 2048, 128]  (token 6144~8191)

环形拓扑:  GPU0 → GPU1 → GPU2 → GPU3 → GPU0 → ...

═══════════════════════════════════════════════════════════════════
Step 0: 用本地 KV 算本地 Q 的部分注意力
═══════════════════════════════════════════════════════════════════

  GPU 0: O₀_partial = softmax(Q₀ @ K₀^T / √128) @ V₀
          m₀ = row_max(Q₀ @ K₀^T / √128)      ∈ [2048]
          l₀ = row_sum(exp(S₀ - m₀))           ∈ [2048]

  GPU 1,2,3: 同理

  此时每 GPU 的 O 只 "看到" 了 2048 个 token → 不完整！

═══════════════════════════════════════════════════════════════════
Step 1: 传 KV + 继续算
═══════════════════════════════════════════════════════════════════

  发送方向: GPU i → GPU (i+1) % 4

  GPU 0 发送 (K₀, V₀) 给 GPU 1
  GPU 1 发送 (K₁, V₁) 给 GPU 2
  GPU 2 发送 (K₂, V₂) 给 GPU 3
  GPU 3 发送 (K₃, V₃) 给 GPU 0

  同时进行：每 GPU 一边发送自己的 KV，一边接收上一 GPU 的 KV

  收到后每 GPU 用新 KV 继续算 O:

  GPU 0: 用 K₃, V₃ (来自 GPU 3) 更新 O₀
          → O₀ 现在 "看到" token 6144~8191 + token 0~2047

  GPU 1: 用 K₀, V₀ (来自 GPU 0) 更新 O₁
          → O₁ 现在 "看到" token 0~2047 + token 2048~4095

  GPU 2: 用 K₁, V₁ (来自 GPU 1) 更新 O₂
  GPU 3: 用 K₂, V₂ (来自 GPU 2) 更新 O₃

  ★ 关键并行 ★：
  当 GPU 0 用 K₃,V₃ 计算注意力时
  → GPU 1 也在同时用 K₀,V₀ 计算注意力
  → GPU 2 也在同时用 K₁,V₁ 计算注意力
  → GPU 3 也在同时用 K₂,V₂ 计算注意力

═══════════════════════════════════════════════════════════════════
Step 2: 继续传 KV
═══════════════════════════════════════════════════════════════════

  GPU 0 发送 (K₃, V₃) [刚从 GPU 3 收到的] 给 GPU 1
  GPU 1 发送 (K₀, V₀) [刚从 GPU 0 收到的] 给 GPU 2
  GPU 2 发送 (K₁, V₁) [刚从 GPU 1 收到的] 给 GPU 3
  GPU 3 发送 (K₂, V₂) [刚从 GPU 2 收到的] 给 GPU 0

  每 GPU 收到新的 KV 块后继续更新 O...

═══════════════════════════════════════════════════════════════════
Step 3: 最后一轮
═══════════════════════════════════════════════════════════════════

  经过 P-1 = 3 轮传递后，每张 GPU 的 KV 都在环上转了一圈
  每 GPU 的 O 都 "看到" 了全部 8192 个 token 的 KV

  最终输出:
  GPU 0: O₀ [1, 2048, 128]  ← Q₀ 对全序列的注意力结果
  GPU 1: O₁ [1, 2048, 128]
  GPU 2: O₂ [1, 2048, 128]
  GPU 3: O₃ [1, 2048, 128]
```

### 7.3 计算-通信重叠的异步流水线

```
这是 Ring Attention 的核心优势 —— 边传边算:

时间 →
┌─────────────────────────────────────────────────────────────────┐
│ GPU 0:                                                          │
│  [计算: Q₀×K₀] [发送 K₀,V₀] [接收 K₃,V₃] [计算: Q₀×K₃]         │
│                          ↕ (重叠)    ↕ (重叠)                     │
│  [通信: 传输中]          [传 K₀V₀]   [传 K₃V₃]                    │
├─────────────────────────────────────────────────────────────────┤
│ GPU 1:                                                          │
│  [计算: Q₁×K₁] [发送 K₁,V₁] [接收 K₀,V₀] [计算: Q₁×K₀]         │
├─────────────────────────────────────────────────────────────────┤
│ GPU 2:                                                          │
│  [计算: Q₂×K₂] [发送 K₂,V₂] [接收 K₁,V₁] [计算: Q₂×K₁]         │
├─────────────────────────────────────────────────────────────────┤
│ GPU 3:                                                          │
│  [计算: Q₃×K₃] [发送 K₃,V₃] [接收 K₂,V₂] [计算: Q₃×K₂]         │
└─────────────────────────────────────────────────────────────────┘

所有 GPU 同时工作，没有哪张在闲着等！
```

### 7.4 Ring Attention 的 Decode 详细流程

```
═══════════════════════════════════════════════════════════════════
          Ring Attention —— Decode 详细流程
═══════════════════════════════════════════════════════════════════

背景:
  已完成 prefill，每 GPU 本地存有各自段的完整 KV Cache
  GPU 0: KV Cache 0  (token 0~2047)
  GPU 1: KV Cache 1  (token 2048~4095)
  GPU 2: KV Cache 2  (token 4096~6143)
  GPU 3: KV Cache 3  (token 6144~8191)

  新生成 1 个 token (假设由 GPU 0 处理其 Q)
  q_new ∈ [1, 1, d]  ← 只有 1 个 token！

═══════════════════════════════════════════════════════════════════
Decode Step: 用新 Q attend 全部 KV Cache
═══════════════════════════════════════════════════════════════════

Step 0: GPU 0 用本地 KV Cache 算部分注意力
  O_new_partial = softmax(q_new @ KV_cache_0^T / √d) @ V_cache_0
  m, l 更新

Step 1: GPU 0 → GPU 1 发送 q_new (而不是全部 KV！)
  ⚡ 优化：Decode 时 Q 只有 1 个 token，传 Q 比传全部 KV 便宜得多！

  GPU 1: 收到 q_new，用本地 KV Cache 1 计算
    O_new_partial += softmax(q_new @ KV_cache_1^T / √d) @ V_cache_1
    (用在线 softmax 合并)

Step 2: GPU 1 → GPU 2 发送 q_new
  GPU 2: 同样计算...

Step 3: GPU 2 → GPU 3 发送 q_new
  GPU 3: 同样计算...

  最后 GPU 3 把完整 O_new 传回 GPU 0
  (或者每 GPU 传自己的部分 O 回 GPU 0 → all-reduce)

==========================
另一种 Decode 方案：传 KV 块而非 Q
==========================

  如果各 GPU 的 KV Cache 不均衡 (新 token 被添加到某张 GPU 上):

  Step i: 每 GPU 向下一 GPU 发送自己新增的 KV Cache 块
          收到后 append 到本地、参与下一 token 的计算
  
  但这通常不如 "传 Q" 高效，因为 KV 比 Q 大得多。

═══════════════════════════════════════════════════════════════════
更实际的 Decode 流程 (所有 GPU 并行生成)
═══════════════════════════════════════════════════════════════════

  如果所有 GPU 各自处理不同的请求 (DP):

  每个 GPU 都有自己请求的新 Q
  获取其他 GPU 的 KV Cache 的方式:
    方案A: 每 GPU 传自己的 KV Cache 块给下一 GPU (传 P-1 轮)
    方案B: 每 GPU 传 Q 给下一 GPU，接收方用本地 KV Cache 计算 O
           然后传 O 回去

  方案B 更优: Q (d维) 通常远小于 KV Cache (S/P × d)
```

### 7.5 Ring Attention 通信量分析

```
Prefill 每轮通信:
  每 GPU 发送: K_block + V_block = 2 × (S/P) × d 个浮点数
  共 P-1 轮
  总通信量/GPU: 2(P-1) × (S/P) × d = 2Sd × (P-1)/P

Decode 通信 (传 Q 方案):
  每 GPU 发送: Q_block = 1 × d 个浮点数
  共 P-1 轮
  总通信量/GPU: (P-1) × d ≈ P × d

  对比 Prefill 通信量: ≈ 2Sd
  Decode 通信量 / Prefill 通信量 ≈ P/(2S) → 极小！
```

## 8. Prefill-Decode 分离 + Ring Attention

> ⭐ **本章为重点章节**。PD 分离是现代推理系统（如 vLLM、SGLang、Sarathi-Serve）的核心架构。

### 8.1 为什么要 PD 分离？

```
Prefill 和 Decode 的计算特性完全不同：

┌──────────────────────────────────────────────────────────────────┐
│  Prefill                              Decode                      │
│  ────────                             ──────                      │
│  ● 计算密集 (矩阵乘法多)               ● 带宽密集 (读 KV Cache)      │
│  ● 延迟不敏感 (几十ms到几s都行)        ● 延迟极度敏感 (每个token都要快)│
│  ● 吞吐优先                            ● 延迟优先                   │
│  ● GPU 算力饱和                         ● GPU 算力大量闲置           │
│                                                                  │
│  问题：如果同一张 GPU 同时做 Prefill 和 Decode：                   │
│  Prefill 的大矩阵乘法会抢占算力 → Decode 延迟飙升 (抖动)            │
│  Decode 的显存带宽被 Prefill 抢占 → token 生成变慢                  │
└──────────────────────────────────────────────────────────────────┘

解决方案: 把 Prefill 和 Decode 分到不同的 GPU/节点上!
```

### 8.2 PD 分离的架构

```
┌──────────────────────────────────────────────────────────────────┐
│                  PD 分离架构全景                                   │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│   请求到达                                                        │
│      │                                                           │
│      ↓                                                           │
│  ┌──────────────────┐                                            │
│  │   调度器/Scheduler │  ← 决定哪个请求去哪                       │
│  └────┬─────────┬───┘                                            │
│       │         │                                                │
│       ↓         ↓                                                │
│  ┌─────────┐  ┌─────────┐                                       │
│  │ Prefill │  │ Decode  │                                       │
│  │ 节点组   │  │ 节点组   │                                       │
│  │         │  │         │                                       │
│  │ GPU 0-3 │  │ GPU 4-7 │                                       │
│  │ (高算力) │  │ (高带宽) │                                       │
│  └────┬────┘  └────┬────┘                                       │
│       │            │                                             │
│       │   KV Cache │                                             │
│       └────────────┘                                             │
│         传输/共享                                                 │
│                                                                  │
│  Prefill 节点: 处理完整 prompt → 生成 KV Cache + 第一个 token      │
│  Decode 节点:  接收 KV Cache → 逐 token 生成 → 更新 KV Cache       │
└──────────────────────────────────────────────────────────────────┘
```

### 8.3 PD 分离的三种模式

```
═══════════════════════════════════════════════════════════════════
模式1: 简单分离 (Splitwise / DistServe)
═══════════════════════════════════════════════════════════════════

  ┌──────────────────┐         ┌──────────────────┐
  │  Prefill 节点     │  KV     │  Decode 节点      │
  │  (GPU 0,1,2,3)   │ ──────→ │  (GPU 4,5,6,7)   │
  │                  │  Cache  │                   │
  │  处理 prompt     │ ←────── │  逐 token 生成    │
  │  产出 KV Cache   │  新 KV  │  追加到 KV Cache  │
  └──────────────────┘         └──────────────────┘

  优点: 简单，Prefill 和 Decode 互不干扰
  缺点: KV Cache 传输量大 (每次 Decode 新增 KV 要同步)

═══════════════════════════════════════════════════════════════════
模式2: Ring Attention PD 分离 (Sarathi-Serve / Mooncake)
═══════════════════════════════════════════════════════════════════

  所有 GPU 都参与 Ring Attention 环
  但不同 GPU 分别负责 Prefill 和 Decode：

  ┌──────────┐    KV     ┌──────────┐    KV     ┌──────────┐
  │ GPU 0    │ ────────→ │ GPU 1    │ ────────→ │ GPU 2    │
  │ Decode   │           │ Prefill  │           │ Decode   │
  │ (KV块0)  │           │ (KV块1)  │           │ (KV块2)  │
  └──────────┘           └──────────┘           └──────────┘
       ↑                                              │
       └──────────────────────────────────────────────┘
                        KV 环传

  Prefill 时:
    每 GPU 用 Ring Attention 传 KV，各自处理序列的一部分
    Prefill GPU 产出新的 KV Cache 块，传给环上的下一个 GPU
    
  Decode 时:
    每 GPU 用 KV Cache 做逐 token 生成
    新生成的 token 的 KV 追加到本地 KV Cache
    通过 Ring 同步 KV Cache 更新

═══════════════════════════════════════════════════════════════════
模式3: 混合 PD 分离 (vLLM v1 架构)
═══════════════════════════════════════════════════════════════════

  Prefill 和 Decode 混在一起，但通过调度器分离资源:

  GPU 0: [40% 算力做 Prefill] [60% 算力做 Decode]
  GPU 1: [40% 算力做 Prefill] [60% 算力做 Decode]
  ...

  通过 CUDA stream 优先级控制，确保 Decode 不受 Prefill 影响
```

### 8.4 PD 分离 + Ring Attention —— Prefill 详细流程

```
═══════════════════════════════════════════════════════════════════
     PD 分离 + Megatron Ring Attention —— Prefill 阶段
═══════════════════════════════════════════════════════════════════

配置:
  8 GPU, 2 个 Prefill GPU (GPU 0,1), 6 个 Decode GPU (GPU 2~7)
  每个 GPU 存一段 KV Cache
  序列 S=16384, d=128, H=32, D=4096

阶段1: 请求路由到 Prefill GPU
═══════════════════════════════════════════════════════════════════

  新请求 prompt (如 4096 tokens) 到达调度器
  
  调度器决定:
    使用 GPU 0 作为主 Prefill GPU
    (或者并行使用多张 Prefill GPU 分段处理)

═══════════════════════════════════════════════════════════════════
阶段2: Prefill GPU 处理新 prompt
═══════════════════════════════════════════════════════════════════

  GPU 0 (Prefill):
    输入: prompt tokens [t₀, t₁, ..., t₄₀₉₅]
    
    Embedding: [1, 4096, 4096]
    
    For each layer:
      ┌────────────────────────────────────────────────────────┐
      │ Attention 计算:                                        │
      │                                                        │
      │ 本地的旧 KV Cache (已有 token):                         │
      │   K_old, V_old ∈ [1, S_local_old, d]                  │
      │                                                        │
      │ 新 prompt 的 K_new, V_new:                             │
      │   K_new, V_new ∈ [1, S_prompt, d]                     │
      │   Q_new ∈ [1, S_prompt, d]                            │
      │                                                        │
      │ 需要 attend 全部 token:                                 │
      │   1. 本地旧 KV Cache                                   │
      │   2. 本地新 KV                                         │
      │   3. ★ 其他 GPU 的 KV Cache (通过 Ring 获取) ★         │
      └────────────────────────────────────────────────────────┘
    
    启动 Ring Attention 环传:
      GPU 0 计算: O_partial = FlashAttention(Q_new, [K_old|K_new], [V_old|V_new])
      然后发送 (K_old|K_new, V_old|V_new) 给 GPU 1 (环中的下一个)
      接收 GPU 7 的 KV Cache 块
      用 GPU 7 的 KV 更新 O_partial
      ... 继续传，直到所有 GPU 的 KV 都被 attend 到

═══════════════════════════════════════════════════════════════════
阶段3: KV Cache 分发
═══════════════════════════════════════════════════════════════════

  Prefill 完成后，新 prompt 的 KV Cache 需要分布到所有 GPU

  方案A: 通过 Ring Attention 的环传过程自然分发
    在 Prefill 的 Ring 计算中，每张 GPU 顺路收下自己负责的那段 KV Cache
    
  方案B: 先由 Prefill GPU 全量计算 KV，再按序列段分发给各 GPU
    GPU 0 算出完整 KV → 按段分发:
      GPU 0 保留: KV[:4096]
      GPU 1 收到: KV[4096:6144]
      GPU 2 收到: KV[6144:8192]
      ...
    
  方案A 更高效 (计算和分布同时完成)

═══════════════════════════════════════════════════════════════════
阶段4: Prefill 完成 → GPU 0 生成第一个 token
═══════════════════════════════════════════════════════════════════

  GPU 0 用完整的全序列注意力结果，通过 LM Head 产出第一个 token
  这个 token 的 KV 追加到 GPU 0 的本地 KV Cache

═══════════════════════════════════════════════════════════════════
阶段5: 状态转移 → 后续 Decode
═══════════════════════════════════════════════════════════════════

  GPU 0 通知调度器: "Prefill 完成, KV Cache 已分布"
  调度器决定后续 Decode 在哪做 (可能是 GPU 0 继续, 也可能转移)
```

### 8.5 PD 分离 + Ring Attention —— Decode 详细流程

```
═══════════════════════════════════════════════════════════════════
     PD 分离 + Megatron Ring Attention —— Decode 阶段
═══════════════════════════════════════════════════════════════════

配置:
  请求已分配在 GPU 2 (Decode GPU)
  各 GPU 已分布存储该请求的 KV Cache
  GPU 0: KV[0:2048]
  GPU 1: KV[2048:4096]
  GPU 2: KV[4096:6144]  ← 主 Decode GPU (持有最新 token)
  GPU 3: KV[6144:8192]
  ...
  GPU 7: KV[14336:16384]

═══════════════════════════════════════════════════════════════════
每轮 Decode Step (生成一个 token)
═══════════════════════════════════════════════════════════════════

Step 1: GPU 2 计算新 token 的 Q
═══════════════════════════════════════════════════════════════════

  前一 token 经过所有层后的 hidden_state
  在 GPU 2 上本地运行:
    q_new = LayerNorm(h) @ W_Q → [1, 1, 32, 128]   (或 TP 分布式)
    k_new = LayerNorm(h) @ W_K → [1, 1, 32, 128]
    v_new = LayerNorm(h) @ W_V → [1, 1, 32, 128]
  追加 k_new, v_new 到 GPU 2 的本地 KV Cache

═══════════════════════════════════════════════════════════════════
Step 2: Ring 传递 Q + 各 GPU 本地 attend (★ 核心！★)
═══════════════════════════════════════════════════════════════════

  方案A: 传 Q 方案 (Decode 推荐！)

  GPU 2 发送 q_new 给 GPU 3 (环中下一节点)
  
  GPU 3: 收到 q_new
    用本地 KV Cache 计算:
      o_partial_3 = FlashAttention(q_new, KV_cache_3.K, KV_cache_3.V)
    发送 q_new 给 GPU 4
    发送 o_partial_3 给 GPU 2

  GPU 4: 同理...
  
  ... 直到 q_new 传回 GPU 2 (经过 GPU 7 → GPU 0 → GPU 1 → GPU 2)

  GPU 2 收集所有 o_partial_i，合并为完整输出:
    用在线 softmax 合并 (和 Ring Attention Prefill 一样)

  ⚡ 通信量分析:
    传 Q: 1个token × H × d = 1 × 32 × 128 = 4096 floats ≈ 16 KB (FP32)
    每 GPU: 收到 Q (16 KB) + 发出 Q (16 KB) + 发回 O (16 KB)
    P 轮: 总通信 ≈ P × 48 KB → 在 NVLink 下可忽略不计

  ⚡ 延迟分析:
    每步: P2P 延迟 × P 跳
    P=8, NVLink P2P 延迟 ~1μs → 总延迟 ~8μs
    对比一次 Decode step 总时间 (~10ms): 通信占比 << 0.1%

  方案B: 传 KV 方案 (少用)

  GPU 2 发送 k_new, v_new 给 GPU 3
  GPU 3: append 到本地 KV Cache
    用本地 Q (无) → 如果有其他请求的 Q 在 GPU 3 上，可以直接用
    发送 (k_new, v_new) 给 GPU 4
  适用于: 多请求共享 KV Cache 的场景

═══════════════════════════════════════════════════════════════════
Step 3: GPU 2 继续完成后续层的计算
═══════════════════════════════════════════════════════════════════

  Attention 输出合并完成 → [1, 1, 32, 128]
  W_O projection → Residual → LayerNorm → MLP
  → 产出下一个 hidden_state

═══════════════════════════════════════════════════════════════════
Step 4: 通过 LM Head 产出 token
═══════════════════════════════════════════════════════════════════

  新 token 的 k, v 追加到 GPU 2 的本地 KV Cache
  重复 Step 1~4 直到生成 EOS 或达到 max_length

═══════════════════════════════════════════════════════════════════
KV Cache 的再平衡
═══════════════════════════════════════════════════════════════════

  问题: GPU 2 一直生成新 token，KV Cache 只在 GPU 2 上增长
  → 不均衡！

  解决方案:
    每生成 K 个 token (如 64)，GPU 2 将新增的 KV 块发给下一个 GPU
    或在 Ring 传 Q 时，顺带把新增 KV 块传递给下一个 GPU

  再平衡流程:
    GPU 2 新增 64 个 token 的 KV 超过阈值
    → 把 KV[4096:4224] 发给 GPU 3 让其存储
    (具体策略取决于 KV Cache 分布策略)
```

### 8.6 PD 分离 + Ring Attention 的 Prefill/Decode 完整张量流动图

```
═══════════════════════════════════════════════════════════════════
                    单层 Attention 张量形状全流程
═══════════════════════════════════════════════════════════════════

假设: 8 GPU 环, S=16384, H=32, d=128, D=4096, TP=1, CP=8
每 GPU 序列段: S/P = 2048

───────────────────────────────────────────────────────────────────
PREFILL 阶段 (处理新 prompt, S_prompt=4096)
───────────────────────────────────────────────────────────────────

GPU 0 (主 Prefill GPU):

  Input: h ∈ [1, 4096, 4096]  (新 prompt tokens)

  Q = h @ W_Q → [1, 4096, 4096] → reshape → [1, 4096, 32, 128] → transpose → [1, 32, 4096, 128]
  K = h @ W_K → [1, 4096, 4096] → reshape → [1, 4096, 32, 128] → transpose → [1, 32, 4096, 128]
  V = h @ W_V → [1, 4096, 4096] → reshape → [1, 4096, 32, 128] → transpose → [1, 32, 4096, 128]

  本地旧 KV Cache: K_old, V_old ∈ [1, 32, 2048, 128]
  合并: K_local = concat([K_old, K], dim=2) → [1, 32, 6144, 128]
        V_local = concat([V_old, V], dim=2) → [1, 32, 6144, 128]

  ┌──── Ring Attention 开始 (Send/Recv 异步流水线) ────┐
  │                                                       │
  │ Round 0: Q × K_local^T                               │
  │   Scores_0 = Q @ K_local^T / √128                   │
  │   → [1, 32, 4096, 6144]                             │
  │   O_0, m_0, l_0 = online_softmax_update(             │
  │        Scores_0, V_local, m_prev=-inf, l_prev=0)     │
  │   → O_0: [1, 32, 4096, 128]                         │
  │                                                       │
  │   发送 K_local, V_local 给 GPU 1                       │
  │   接收 GPU 7 的 KV Cache 块                            │
  │                                                       │
  │ Round 1: Q × K_from_GPU7^T  [1, 32, 4096, 2048]     │
  │   O_1 = online_softmax_update(Scores_1, V_from_GPU7,  │
  │                                m_0, l_0)              │
  │   发送 GPU 7 的 KV 给 GPU 1                            │
  │   接收 GPU 6 的 KV Cache 块                            │
  │                                                       │
  │ Round 2~6: 同理...                                    │
  │                                                       │
  │ Round 7 (最后一轮): 收到 GPU 1 传来的原始 GPU 1 的 KV  │
  │   O_final = [1, 32, 4096, 128]  (完整的全序列注意力)  │
  └───────────────────────────────────────────────────────┘

  Output: reshape O_final → [1, 4096, 4096] → W_O → [1, 4096, 4096]

  同时，KV Cache 分发:
    GPU 0 保留: K_new[0:2048], V_new[0:2048]
    GPU 1 分到: K_new[2048:4096], V_new[2048:4096]
    其余 GPU: 本次没分到新 KV (旧的 KV 已有分布)

───────────────────────────────────────────────────────────────────
DECODE 阶段 (生成 token t_{i+1})
───────────────────────────────────────────────────────────────────

  GPU 2 (Decode GPU, 持有 KV[4096:6144] 段):

  Input: h ∈ [1, 1, 4096]  (上一 token 的 hidden state)

  Q = h @ W_Q → [1, 1, 32, 128]
  K = h @ W_K → [1, 1, 32, 128]  ← 追加到本地 KV Cache
  V = h @ W_V → [1, 1, 32, 128]  ← 追加到本地 KV Cache

  GPU 2 本地 KV Cache (含新 token): K_2, V_2 ∈ [1, 32, 2049, 128]

  ★ Decode Ring Attention (传 Q 方案) ★

  GPU 2 本地计算:
    O_local = FlashAttention(Q, K_2, V_2)
    → [1, 1, 32, 128], m_2, l_2
    
  发送 Q 给 GPU 3
  GPU 3: O_3 = FlashAttention(Q, K_3, V_3) → 发 O_3 回 GPU 2 → 发 Q 给 GPU 4
  ...

    O_final = online_softmax_merge([O_local, O_3, O_4, ..., O_7, O_0, O_1])
    → [1, 1, 32, 128]

  reshape → [1, 1, 4096] → W_O → [1, 1, 4096]
  → Residual → LN → MLP → [1, 1, 4096]
  → LM Head → 下一个 token
```

### 8.7 PD 分离 + Ring Attention 的优化技术

#### 8.7.1 KV Cache 分层存储

```
┌─────────────────────────────────────────────────────────────────┐
│  热层 (GPU 显存): 最近 K 个 token 的 KV Cache                    │
│  温层 (CPU 内存): 较早的 token 的 KV Cache                       │
│  冷层 (SSD): 最久远的 token 的 KV Cache                         │
│                                                                  │
│  Decode 时，Q 优先 attend 热层 (命中率 > 95%)                     │
│  少量 Q 需要 attend 温/冷层时，异步加载，不阻塞 Decode            │
└─────────────────────────────────────────────────────────────────┘
```

#### 8.7.2 计算-通信-调度三重重叠

```
时间 →
┌──────────────────────────────────────────────────────────────────┐
│ GPU 0 (Decode):                                                  │
│  [send Q] [本地 attn] [recv partial O] [merge O] [MLP] [LM Head] │
│      ↓         ↓            ↓                                    │
│  [Ring 传Q]  [FlashAttn]  [传输重叠]                              │
├──────────────────────────────────────────────────────────────────┤
│ GPU 1 (Prefill):                                                 │
│  [recv prompt] [QKV proj] [Ring 传 KV] [本地 attn] [recv KV]     │
│                                                 ↓                │
│                                          [计算和通信重叠]          │
└──────────────────────────────────────────────────────────────────┘
```

#### 8.7.3 请求级调度 (Request-Level Scheduling)

```
调度器维护:
  1. Prefill 队列: 按到达顺序 + 长度分桶 (防止长 prompt 饿死短的)
  2. Decode 队列: 按优先级 (延迟敏感度)
  3. 混合队列: Prefill 和 Decode 共享 GPU 时的 slot 分配

当 Decode GPU 空闲时:
  可以临时接受短的 Prefill 请求 (利用空闲算力)
  但不接受长 Prefill → 会阻塞其他请求的 Decode
```

### 10.1 通信量公式速查

| 技术 | 单 GPU 显存 (注意力部分) | 每层通信量 / GPU | 通信轮数 |
|------|------------------------|------------------|---------|
| 无并行 | `O(B × H × S²)` | 0 | 0 |
| TP (P路) | `O(B × H × S²)` (注意力不省) | `4 × B × S × D` (all-reduce) ×2 | 2 |
| Ring Attention (P路) | `O(B × H × (S/P)²)` | `2(P-1) × (S/P) × d × 2` (传 KV, Fwd) | P-1 |
| Ulysses (P路) | `O(B × H/P × S²)` | `4 × B × S × D / P` (all-to-all) | 2 |
| TP+Ulysses (TP×CP) | `O(B × H/(TP×CP) × S²)` | TP: `4×B×S×D/TP` + CP: `4×B×S×D/(TP×CP)` | TP:2 + CP:2 |

> 注: Ring Attention 的 P-1 轮可以流水线重叠；Ulysses 的 2 轮必须串行。

### 10.2 在线 Softmax 算法

```python
# 数值稳定的分块 softmax，是所有分布式注意力的数学基础

def online_softmax_update(S_new, V_new, m_old, l_old):
    """
    S_new: 当前块的注意力分数 [B, H, Q_len, KV_block_len]
    V_new: 当前块的 V [B, H, KV_block_len, d]
    m_old, l_old: 之前所有块的 running max 和 running sum
    
    返回: O_partial, m_new, l_new
    """
    m_new = max(m_old, S_new.max(dim=-1))  # [B, H, Q_len]
    
    # 数值稳定地合并
    l_new = exp(m_old - m_new) * l_old + \
            exp(S_new - m_new.unsqueeze(-1)).sum(dim=-1)  # [B, H, Q_len]
    
    P_new = exp(S_new - m_new.unsqueeze(-1))  # [B, H, Q_len, KV_block_len]
    P_new = P_new / l_new.unsqueeze(-1)
    
    # 更新之前积累的输出
    O_old_scaled = diag(exp(m_old - m_new)) @ O_old  # 用新 max 重新缩放
    O_new_contrib = P_new @ V_new  # [B, H, Q_len, d]
    O_new = O_old_scaled + O_new_contrib
    
    return O_new, m_new, l_new
```

### 10.3 All-to-All 通信通讯量推导

```
设 CP 并行度为 P，每 GPU 序列长度为 S_local = S/P

第一轮 All-to-All (序列→head):
  每 GPU 发送: B × S_local × H × d × 3 (QKV)
  由于 all-to-all 均匀分散到 P 个 GPU:
    每 GPU 实际发出: (B × S_local × H × d × 3) × (P-1)/P
    每 GPU 实际收到: (B × S_local × H × d × 3) × (P-1)/P
  总通信量/GPU ≈ B × S_local × D × 3 (P 大时近似)
                  = 3 × B × S × D / P

第二轮 All-to-All (head→序列):
  每 GPU 发送: B × S × (H/P) × d (输出 O)
  分散到 P 个 GPU:
    每 GPU 收发: B × S × (H/P) × d × (P-1)/P
  总通信量/GPU ≈ B × S × D / P

两轮总计: ~4 × B × S × D / P  float
        ≈ 4 × B × S × D × 2 / P  bytes (FP16/BF16)
```

| **RDMA** | Remote Direct Memory Access | 跨节点直接内存访问 |

| **HBM** | High Bandwidth Memory | GPU 显存 |
| **USP** | Unified Sequence Parallelism | 统一序列并行 (Ulysses + Ring 混合) |
| **D/P (D over P)** | Hidden / Parallelism degree | D 维切到 P 个 GPU 上 |
