## 第一章：什么是语音生成

### 1.1 一句话讲清楚

**语音识别（ASR）是听人说话，语音生成是替人说话。**

```
ASR:   音频 🔊 → 模型 → 文字 📝     （你在语音识别教材里学的内容）
TTS:   文字 📝 → 模型 → 音频 🔊     （这本教材的内容）
```

### 1.2 传统 TTS 和现代语音生成的区别

#### 传统 TTS（Text-to-Speech）

最早的"文字转语音"技术分成两步：

```
文字 → 语言分析（分词、注音、韵律预测）→ 声学模型 → 波形合成 → 音频
```

传统方法的问题：声音像机器人（"Siri 腔"），因为每一步都是规则驱动的，很难自然地模仿人类的语气、停顿和情感。

#### 现代语音生成（基于大模型）

现代方法借鉴了 LLM（大语言模型）的思路：

```
文字 → 文本token → [Text-Speech LM] → 语音token → [解码器] → 音频
```

核心变化：用一个**语言模型**来统一处理"文字→声音"的映射，就像 GPT 能续写文字一样，这个模型能"续写"语音。生成的语音更自然、更有表现力。

### 1.3 这本教材的主线

和语音识别教材以"Embedding"为主线一样，这本教材以**"Speech Tokenizer"**为主线：

```
声音是怎么被压缩成离散 token 的？
  → 这些离散 token 长什么样？
    → 训练时和推理时，token 化流程有什么不同？
      → 最后模型怎么从 token 还原出声音？
```

## 第二章：Speech Tokenizer——音频版的"分词器"

### 2.1 为什么需要 Speech Tokenizer

> 📖 **前置概念**：在语音识别教材的第二章讲过——文本 token 化就是把一句话切成离散的符号（词/字素/音素）。LLM 里每个 token 是一个数字 ID，模型预测下一个 token ID。

语音生成面临同样的问题：如果想让一个语言模型"说出"语音，语音必须先被转成**离散的 token**，语言模型才能像处理文字一样处理语音。

**Speech Tokenizer 就是做这件事的——把连续的音频压缩成离散的 token 序列。**

```
为什么不能直接用原始波形？

一段 3 秒 16kHz 音频 = 48000 个浮点数
如果每个采样点是一个"token"，那 3 秒就有 48000 个 token
→ 语言模型要预测 48000 步才能生成 3 秒语音
→ 太慢了，不可行

Speech Tokenizer 的目标：
  48000 个采样点 → 压缩成 ~150 个离散 token
  → 每个 token 代表约 20ms 的音频
  → 语言模型只需要 150 步就能生成 3 秒语音
```

### 2.2 Speech Tokenizer 的两半——Encoder 和 Decoder

Speech Tokenizer 是一个"编码器-解码器"结构，和自编码器（Autoencoder）思路一样：

```
┌──────────────────────────────────────────────────────────────┐
│                    Speech Tokenizer                           │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│   原始音频 [48000]                                            │
│       ↓                                                       │
│   ┌──────────────────────┐                                    │
│   │  Encoder（编码器）   │  ← 把音频压缩成连续的特征向量      │
│   │  卷积 + 降采样       │                                    │
│   └──────────┬───────────┘                                    │
│              ↓                                                │
│   连续特征 [T, 128]                                           │
│       ↓                                                       │
│   ┌──────────────────────┐                                    │
│   │  量化器（Quantizer） │  ← 把连续向量变成离散token ← 核心！│
│   │  FSQ / VQ / RVQ      │                                    │
│   └──────────┬───────────┘                                    │
│              ↓                                                │
│   离散 token 序列: [142, 7, 891, 56, 230, ...]               │
│              ↓  （这是语言模型要学的部分）                     │
│              ↓  （如果你想还原声音，继续往下走）               │
│   ┌──────────────────────┐                                    │
│   │  Decoder（解码器）   │  ← 把离散token还原成音频波形       │
│   │  反卷积 + 上采样     │                                    │
│   └──────────┬───────────┘                                    │
│              ↓                                                │
│   重建音频 [48000]（和原始音频尽可能像）                       │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### 2.3 两个关键术语

| 术语 | 解释 | 类比 |
|------|------|------|
| **连续特征（Continuous Features）** | Encoder 输出的浮点数向量，每个值可以是任意实数 | 一张彩色照片（颜色是连续的） |
| **离散 Token（Discrete Tokens）** | 量化器输出的整数 ID，每个值只能是有限的几个选择之一 | 一个调色板编号（"3号色"——只有几个固定颜色） |

> **核心问题**：把连续的特征向量变成离散 token，就像把"天蓝色"归类到最近的调色板颜色"蓝色"。"归类"这个操作就是**量化（Quantization）**——第四章会详细讲。

### 2.4 Speech Tokenizer 的训练目标——重建得越像越好

Speech Tokenizer 训练时不涉及文字——它只是学着把音频压缩再还原：

```
训练目标：重建音频 ≈ 原始音频

损失函数：通常是重建误差（MSE / L1）+ 对抗损失（让解码出来的声音更"真实"）
```

训练完成后，Encoder + 量化器 用来把音频转成 token（给语言模型做训练数据），Decoder 用来把 token 还原成音频（推理时用）。

## 第三章：从声波到连续特征——Encoder 的内部过程

> 📖 **前置概念**：这一章涉及分帧、DFT、梅尔滤波、卷积降采样，和语音识别教材的第三、四、五、七章内容对应。如果某一步看不懂，去翻对应的章节。

### 3.1 你笔记里的维度 `[1, 帧数, 128]` 是什么意思

这是 Speech Tokenizer 的 Encoder 输出——**还没经过量化的连续特征**：

```
[1, T, 128]
 ↑   ↑    ↑
 │   │    └── 每帧的特征维度 = 128 维
 │   └─────── 时间帧数（取决于音频长度）
 └─────────── batch size = 1（一次处理一条音频）
```

**这个 `[1, T, 128]` 是怎么来的？** 下面逐步拆解。

### 3.2 从原始波形的维度变化

以一段 3 秒 16kHz 音频为例：

```
第 0 步：原始音频
  采样率: 16000 Hz
  时长: 3 秒
  采样点数: 3 × 16000 = 48000
  
  shape: [1, 48000]     ← batch=1, 采样点数


第 1 步：分帧 + Fbank（和 ASR 一模一样）
  帧长 25ms → 每帧 400 个采样点 (@16kHz)
  帧移 10ms → 每 10ms 产生一帧
  梅尔滤波器: 80 个
  
  生成帧数 ≈ (48000 - 400) / 160 + 1 ≈ 299 帧
  
  shape: [1, 299, 80]   ← batch=1, 299帧, 80维Fbank


第 2 步：Encoder 卷积层 + 降采样
  Speech Tokenizer 的 Encoder 会对 Fbank 特征做多层卷积，
  每一步在时间维度降采样，同时扩展特征维度。
  
  具体过程（以 SEED 或类似模型为例）：
  
  输入: [1, 299, 80]
    ↓ ConvBlock 1 (stride=2, channels=128)
    [1, 150, 128]      ← 时间减半，维度变成128
    ↓ ConvBlock 2 (stride=2, channels=128)
    [1, 75, 128]       ← 时间再减半，维度保持128
    ↓ ConvBlock 3 (stride=1, channels=128)  
    [1, 75, 128]       ← stride=1不改变时间维度，只做特征变换
  
  Encoder 最终输出 shape: [1, 75, 128]
                            ↑   ↑    ↑
                         batch 帧数  特征维度
```

### 3.3 为什么是 128 维

128 是 Encoder 输出维度的一个**设计选择**（超参数），不是什么神秘数字。不同的 Speech Tokenizer 实现会用不同的值：

| 模型 | Encoder 输出维度 | 说明 |
|------|-----------------|------|
| EnCodec (Meta) | 128 | 常用的开源 codec |
| SpeechTokenizer | 128 | 和 EnCodec 类似 |
| DAC (Descript Audio Codec) | 512 或 1024 | 更高维度，更高音质 |
| CosyVoice / SEED | 512（投影后） | 阿里方案，下面会讲 |

### 3.4 你笔记里"维度从 128 变到 512"是怎么回事

这是关键的一步。Encoder 输出是 128 维，但在送入量化器之前，通常会**加一个线性投影层**把维度放大：

```
Encoder 输出: [1, 75, 128]
  ↓ 线性投影（Linear Projection）
  ↓ 128 → 512
投影后: [1, 75, 512]     ← 更高维的特征空间
  ↓ 送入量化器（FSQ / VQ）
```

**为什么要放大到 512？**

量化是一个"有损"操作——把一个连续的 512 维向量映射到一个离散的整数。维度越高，向量空间越"宽敞"，不同声音的向量就能分得越开，量化造成的精度损失就越小。

```
类比：

128 维空间：像一个 10 平方米的房间
  → 塞进去 1000 个"声音点"，它们挤在一起
  → 量化时容易分不清"这个是 A 类还是 B 类"

512 维空间：像一个 100 平方米的大厅
  → 同样的 1000 个"声音点"，彼此分得很开
  → 量化时容易精确归类
```

> **一句话**：128 → 512 是一个线性投影，目的是把特征映射到更高维空间，为后面的量化步骤提供更丰富的表示。

## 第四章：FSQ 量化——把连续向量变成离散编号

> 📖 **前置概念**：本章新概念集中。**量化（Quantization）** 就是把连续值映射到有限个离散值的过程。语音识别教材里讲过"采样量化"（把连续幅度变成 16 bit 整数），那是信号的量化；这里是对**特征向量**的量化。原理相通，对象不同。

### 4.1 什么是量化——最直观的解释

想象你要把世界上所有的颜色分类。世界上的颜色是**无限的**（连续），但你说"这是红色"时，你把无限种红色归成了一类（离散）。

```
连续的颜色光谱：
  胭脂红、朱红、砖红、绯红、酒红、玫瑰红、樱桃红……
  无穷无尽，每一个都不一样

离散的颜色标签：
  红色、蓝色、绿色、黄色——只有 4 个编号，0, 1, 2, 3

量化就是：把每一个连续的颜色值，归类到它最接近的标签编号。
  "胭脂红" → 编号 0（红色）
  "天蓝色" → 编号 1（蓝色）
```

音频量化也是一样：把一个连续的特征向量，归类到"最接近的码字编号"。

### 4.2 两种主流量化方法——VQ 和 FSQ

#### VQ（Vector Quantization，向量量化）

```
┌───────────────────────────────────────┐
│  维护一个"码本"（Codebook）：          │
│                                        │
│  码字 0: [0.23, -0.15, 0.87, ...,  ]  │
│  码字 1: [-0.56, 0.34, -0.12, ..., ]  │
│  码字 2: [0.78, 0.01, -0.45, ..., ]   │
│  ...                                   │
│  码字 1023: [0.12, -0.67, 0.34, ...,] │
│                                        │
│  码本大小 = 1024（有 1024 个码字）     │
│  每个码字是一个 512 维向量             │
│  码本是**可学习的参数**                │
└───────────────────────────────────────┘

输入一个向量 → 和码本里所有码字算距离 → 选最近的 → 输出该码字的编号

例如：
  输入: [0.25, -0.13, 0.85, ...,]
  距离码字0: 0.05  ← 最近！
  距离码字1: 1.23
  ...
  → 输出编号: 0
```

VQ 的问题是：**需要维护一个大码本**（比如 1024 个 512 维向量），参数多，训练不稳定（"码本崩塌"——很多码字永远用不到）。

#### FSQ（Finite Scalar Quantization，有限标量量化）

FSQ 不维护一个可学习的码本，它用一个非常巧妙的思路避开了码本：**对向量的每一维独立量化，取每一维的 L 个等距值，所有维度的排列组合就是"隐式码本"。**

```
VQ 的做法：                FSQ 的做法：
维护 1024 个码字向量      不维护码本！
输入向量和每个码字         把向量的每一维分别量化
比较距离                   到 L 个固定级别
                          "码本" = 所有级别组合的
                          笛卡尔积

类比：
VQ = 1024 张色卡           FSQ = "R通道分5档 × G通道分5档 × B通道分5档
      每张都不一样               = 125种颜色"
      需要记住每张长什么样       不需要记色卡，只需要知道5档
                               每档的数值
```

### 4.3 FSQ 的具体过程（你笔记的核心）

以你笔记里的情况为例：Encoder 输出投影到 512 维，FSQ 对这 512 维做量化。

#### 4.3.1 FSQ 的超参数设置

FSQ 不是对 512 维全部做量化——那样组合数太大了。它是把 512 维**分成若干组**，每组分别量化：

```
常见的 FSQ 配置：

方案1（简单）：
  512 维 → 分成 64 组，每组 8 维
  每组 8 维，每维 3 个级别 → 每组有 3⁸ = 6561 个可能的值
  一共 64 组 → 64 个 token

方案2（更常用）：
  512 维 → 分成 8 组，每组 64 维  
  每组 64 维，每维 5 个级别 → 每组有 5⁶⁴ = 天文数字个可能的值
  → 不可行！

所以实际上 FSQ 的分组是这样的：
  512 维 → 分成 N 组，每组 d 维，每维 L 个级别
  每组内部是完整的 L^d 组合
  关键参数：d（每组维度）要小，L（级别数）也要小
```

**FSQ 的核心超参数**：

| 参数 | 含义 | 典型值 | 对效果的影响 |
|------|------|--------|-------------|
| d | 每组内的维度数 | 3~8 | d 越小，每组 token 越少，但信息越碎片化 |
| L | 每维的级别数 | 3~5 | L 越小，量化越粗糙，但 token 种类越少 |
| 组数 | 512 / d | 取决于 d | 组数 = 每秒输出的 token 数（约等于帧数） |

#### 4.3.2 一组内的量化过程（核心算法）

以 **d=5, L=3** 为例，取 512 维中的连续 5 维来做量化：

```
输入：一个 5 维向量（从 512 维中切出来的一小段）
  v = [0.72, -1.35, 0.18, 2.41, -0.63]

第 1 步：将每个值约束到 [-1, 1] 范围
  用 tanh 或 clamp 函数

  v_clamped = [0.72, -1.0, 0.18, 1.0, -0.63]
               ↑      ↑     ↑     ↑     ↑
             不变   超过-1  不变  超过1  不变
                        被截断        被截断


第 2 步：映射到 L 个离散级别

  L=3 时，级别是：-1, 0, 1（均匀分布在 [-1, 1] 之间）

  规则：把 [-1, 1] 分成 L-1=2 段 → 每段宽度 = 2/2 = 1.0

  具体的量化映射：
    值在 [-1.0, -0.5) → 量化到 -1
    值在 [-0.5,  0.5) → 量化到  0
    值在 [ 0.5,  1.0] → 量化到  1

  v_clamped = [0.72, -1.0, 0.18, 1.0, -0.63]

  维度 0: 0.72  → 在 [0.5, 1.0] → 量化值 = 1
  维度 1: -1.0  → 在 [-1.0, -0.5] → 量化值 = -1
  维度 2: 0.18  → 在 [-0.5, 0.5] → 量化值 = 0
  维度 3: 1.0   → 在 [0.5, 1.0] → 量化值 = 1
  维度 4: -0.63 → 在 [-1.0, -0.5] → 量化值 = -1

  v_quantized = [1, -1, 0, 1, -1]


第 3 步：将离散值序列转换成唯一的编号

  每维有 L=3 种可能值（-1, 0, 1），5 维的组合总数 = 3⁵ = 243
  
  给每个组合分配一个唯一编号：
  
  [1, -1, 0, 1, -1]  的编号 = ?
  
  计算规则（类似进制转换）：
    先把 [-1, 0, 1] 映射成 [0, 1, 2]
    → [1, -1, 0, 1, -1] 变成 [2, 0, 1, 2, 0]
    
    编号 = 2×3⁴ + 0×3³ + 1×3² + 2×3¹ + 0×3⁰
         = 2×81 + 0×27 + 1×9 + 2×3 + 0×1
         = 162 + 0 + 9 + 6 + 0
         = 177
    
  → 这 5 维被量化成一个 FSQ token: 177


第 4 步：解码（从 token 编号还原连续向量）

  编号 177 → 进制分解：
    177 ÷ 3 → 商 59, 余 0（最低位 → 维度4 = 0 → -1）
    59 ÷ 3  → 商 19, 余 2（→ 维度3 = 2 → 1）
    19 ÷ 3  → 商 6,  余 1（→ 维度2 = 1 → 0）
    6 ÷ 3   → 商 2,  余 0（→ 维度1 = 0 → -1）
    2 ÷ 3   → 商 0,  余 2（→ 维度0 = 2 → 1）
    
    → [2, 0, 1, 2, 0] → 映射回 [-1, 0, 1]
    → [1, -1, 0, 1, -1]
    
  这就是"量化后的连续向量"（虽然它的值仍然是离散的 -1, 0, 1，
  但在 Decoder 看来它就是近似原向量的连续值）
```

### 4.4 完整 512 维的 FSQ 处理

把 512 维分成所有组，每组独立做同样的操作：

```
[1, 75, 512]     ← 经过投影的 Encoder 输出

按最后一个维度分组：
  维 0-4:    第 1 组 (d=5) → FSQ → token ID₁   (0 ~ 3⁵-1=242)
  维 5-9:    第 2 组 (d=5) → FSQ → token ID₂
  维 10-14:  第 3 组 (d=5) → FSQ → token ID₃
  ...（如果不整除，最后一组小于 d 维也照做）
  维 508-512: 第 N 组        → FSQ → token ID_N

结果：
  - 如果 d=5, N = 512/5 ≈ 102 组（每组独立量化）
  - 每帧输出 102 个离散 token
  - 75 帧 × 102 token/帧 ≈ 7650 个 token
```

**等等——这个 token 数太多了！** 75 帧 × 102 token = 7650 token 代表 3 秒音频，每秒 ~2500 token。语言模型每秒要预测 2500 次——这在工程上不太可行。

实践中，FSQ 通常不是按"帧"来分组的，而是把**整个特征映射压缩到较少的维度**再做量化：

```
实际做法：

先降维： [1, 75, 512] → reshape/project → [1, token_count, d]
                                        ↑
                                  比如 [1, 12, 8]
                                  12 个 token，每个 8 维

然后对每个 8 维 token 做 FSQ：
  d=8, L=3 → 每组码本大小 3⁸ = 6561
  → 每 3 秒 12 个 token × 75 帧 = 900 个 token
  → 每秒 300 个 token（好多了！）
```

### 4.5 FSQ 的超参数怎么选——对最终效果的影响

| 超参数调整 | 效果 | 适用场景 |
|-----------|------|---------|
| L 增大（如 3→5） | 量化更精细，但码本更大（d=5：L=3→243种, L=5→3125种） | 高质量语音生成 |
| L 减小（如 3→2） | 量化更粗糙，码本更小 | 低比特率传输 |
| d 增大（如 5→8） | 每组 token 表达能力更强，但码本爆炸（3⁸=6561） | 需要更大的 LM 词表 |
| d 减小（如 5→3） | 每组 token 更简单，但需要更多组 | 小模型场景 |

> **FSQ 的码本是"隐式的"**——不需要存储 6561 个向量，只需要知道 d=5, L=3 就能随时计算出任何编号对应的向量。这是 FSQ 相比 VQ 的最大优势：**没有"码本崩塌"问题，参数几乎为零。**

### 4.6 FSQ 的输入输出总结

```
┌──────────────────────────────────────────────────┐
│              FSQ 量化——输入输出一览                │
├──────────────────────────────────────────────────┤
│                                                   │
│  输入:                                             │
│    连续特征向量，shape: [batch, num_tokens, d]     │
│    例如: [1, 12, 8]                                │
│    （12 个 token，每个 8 维浮点数向量）             │
│                                                   │
│  超参数:                                           │
│    d = 8（每组维度）                               │
│    L = 3（每维级别数）                             │
│    码本大小 = L^d = 3⁸ = 6561                     │
│                                                   │
│  处理过程:                                         │
│    对每组 8 维向量 → 每维 clamp 到 [-1,1]         │
│    → 每维量化到 L 个级别 → 组合编号 0~L^d-1       │
│                                                   │
│  输出:                                             │
│    离散 token ID: [1, 12]  ← 整数，如 [142, 3281,│
│                              55, 6100, ...]       │
│    量化后向量: [1, 12, 8]   ← 用于 Decoder 重建   │
│                                                   │
│  训练时: 量化后向量 → Decoder → 重建音频           │
│          (计算重建 loss，梯度通过 straight-through │
│           estimator 跳过量化步骤回传)              │
│                                                   │
│  推理时: 离散 token ID → 查隐式码本 → 量化后向量   │
│          → Decoder → 音频                          │
│                                                   │
└──────────────────────────────────────────────────┘
```

## 第五章：训练与推理——预处理流程的对比

> 📖 **这是你笔记里特别关注的问题**："搞清楚音频生成的数据预处理，包括训练阶段和推理阶段的预处理，以及他们之间有什么区别"。

### 5.1 训练阶段的预处理

训练阶段的目标：**给 Text-Speech LM 准备训练数据**。需要把海量音频通过 Speech Tokenizer 转成离散 token 序列。

```
训练预处理的完整流程：

┌─────────────────────────────────────────────────────────────┐
│  原始音频 + 对应文字                                          │
│      ↓                                                       │
│  音频 → Speech Tokenizer Encoder → 连续特征 → FSQ量化       │
│      ↓                                                       │
│  得到：离散 token 序列                                       │
│  例如：[142, 8921, 55, 6100, 231, 88, 3401, ...]            │
│      ↓                                                       │
│  文字 → Text Tokenizer（BPE/字级别）→ 文本 token 序列        │
│  例如：[<text_start>, 我, 今天, 去, 学校, <text_end>]       │
│      ↓                                                       │
│  拼接成训练样本：                                             │
│  [<text_start>, 我, 今天, 去, 学校, <text_end>,              │
│   <speech_start>, 142, 8921, 55, 6100, 231, ..., <speech_end>]│
│      ↓                                                       │
│  喂给 Text-Speech LM 做 next-token prediction                 │
│  （模型学会：看到文字 → 预测对应的语音 token）                │
└─────────────────────────────────────────────────────────────┘
```

#### 训练预处理的关键特点

- **音频和文字都要处理**：文字转文本 token，音频转语音 token
- **只需要做一次**：所有训练数据处理完后存下来，训练时直接加载
- **Encoder 是冻结的**：Speech Tokenizer 已经预训练好了，训练 Text-Speech LM 时 Encoder 不更新参数
- **不需要 Decoder**：训练阶段只用 Speech Tokenizer 的 Encoder + 量化器来获取 token，Decoder 留着推理时用

### 5.2 推理阶段的预处理

推理阶段的目标：**把用户的输入文字转成初始 token 序列，喂给 Text-Speech LM 生成语音 token，再用 Decoder 还原成音频。**

```
推理的完整流程：

┌─────────────────────────────────────────────────────────────┐
│  用户输入文字: "我今天去学校"                                 │
│      ↓                                                       │
│  Text Tokenizer → 文本 token 序列                            │
│  [<text_start>, 我, 今天, 去, 学校, <text_end>]             │
│      ↓                                                       │
│  喂给 Text-Speech LM                                         │
│      ↓                                                       │
│  Text-Speech LM 自回归生成语音 token                         │
│  生成的序列: [<speech_start>, 142, 8921, 55, 6100, 231, ...] │
│      ↓                                                       │
│  FSQ 逆量化：离散 token → 量化后向量                         │
│  142 → [1, -1, 0, 1, -1, 0, 1, 0]（查隐式码本）            │
│  8921 → [...]                                                │
│  ...                                                          │
│  所有向量拼接: [N, 8]（N 个 token，每个 8 维）               │
│      ↓                                                       │
│  Speech Tokenizer Decoder                                    │
│  上采样 + 反卷积 → 波形                                      │
│      ↓                                                       │
│  输出音频: 🔊 "我今天去学校"                                 │
└─────────────────────────────────────────────────────────────┘
```

#### 推理预处理的关键特点

- **只有文字输入**：不需要处理音频（因为音频是我们要生成的！）
- **Encoder 完全不用**：推理时只需要 Decoder
- **需要逆量化步骤**：模型输出的离散 token 要先还原成连续向量，Decoder 才能处理

### 5.3 训练 vs 推理——核心区别一览

```
               训练阶段                        推理阶段
              ┌──────────────┐              ┌──────────────┐
输入:          │ 音频+文字     │              │ 只有文字      │
              ├──────────────┤              ├──────────────┤
Speech        │ Encoder ✅   │              │ Encoder ❌   │
Tokenizer:    │ FSQ量化 ✅   │              │ FSQ量化 ❌   │
              │ Decoder ❌   │              │ FSQ逆量化 ✅ │
              │              │              │ Decoder ✅   │
              ├──────────────┤              ├──────────────┤
输出:          │ token序列    │              │ 音频          │
              │ (给LM训练用) │              │ (给用户听)    │
              ├──────────────┤              ├──────────────┤
数据流:        │ 音频→token   │              │ token→音频   │
              │ (编码方向)    │              │ (解码方向)    │
              └──────────────┘              └──────────────┘
```

```
直观理解：

  训练时：你有一堆"原音 + 字幕"的视频
    → 把原音转成 token（像把 MP3 压成 ZIP）
    → 让模型学"看到字幕 → 预测这段 ZIP 内容"

  推理时：你告诉模型"下面是一段说'我今天去学校'的音频："
    → 模型自己生成 ZIP 内容
    → 把 ZIP 解压回音频波形
```

> **一句话**：训练时走"音频→token"方向，推理时走"token→音频"方向。Speech Tokenizer 的 Encoder + 量化器管编码，Decoder 管解码。Text-Speech LM 夹在中间，负责"从文字 token 预测语音 token"。

## 第六章：Text-Speech LM 简介和完整回顾

> 📖 你笔记里提到"text speech lm 是下会需要讲的内容，开了个头简单讲了一下总体的过程"。这一章做一个扼要的概述，为下次学习搭好框架。

### 6.1 Text-Speech LM 做什么

它和 GPT 做的是同一件事——**预测下一个 token**——区别在于 GPT 预测下一个文字 token，Text-Speech LM 预测下一个**语音 token**：

```
GPT:
  输入: "今天天气真" → 预测: "好"
  输入: "今天天气真好" → 预测: "，"

Text-Speech LM:
  输入: [<text_start>, 我, 今天, 去, 学校, <text_end>, <speech_start>, 142, 8921]
  → 预测下一个语音 token: 55
  
  输入: [..., 142, 8921, 55]
  → 预测: 6100
  
  ...直到预测出 <speech_end>
  → 收集所有语音 token → 送 Decoder → 变成音频
```

### 6.2 为什么用 LM 架构做语音生成

LLM 已经证明了大语言模型的强大能力（理解、推理、生成）。语音生成借用同一套基础设施：

```
LLM（文本）:               Text-Speech LM:

Token ID → Embedding       Token ID → Embedding
    ↓                          ↓
Transformer Decoder        Transformer Decoder
  (self-attention)           (self-attention)
    ↓                          ↓
预测下一个 Token ID         预测下一个 Token ID

唯一区别：词表不同
  LLM:      词表 = 文字token（"我","今","天"...共~50k个）
  Text-Speech LM: 词表 = 文字token ∪ 语音token
                语音词表大小 = FSQ码本大小 = L^d（如 3⁸=6561）
```

### 6.3 完整技术栈回顾

把整本教材串起来，语音生成大模型的完整链路：

```
┌──────────────────────────────────────────────────────────────┐
│              现代语音生成大模型完整链路                         │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  ┌───────────────────────────────────────────────────────┐   │
│  │              Speech Tokenizer（离线预训练）             │   │
│  │                                                        │   │
│  │  音频 → Encoder → [1,T,128] → 投影 → [1,T,512]        │   │
│  │               → FSQ量化 → 离散token                    │   │
│  │                                                        │   │
│  │  离散token → FSQ逆量化 → 向量 → Decoder → 音频         │   │
│  └───────────────────────────────────────────────────────┘   │
│                          ↓                                    │
│  ┌───────────────────────────────────────────────────────┐   │
│  │              Text-Speech LM（主训练）                   │   │
│  │                                                        │   │
│  │  文本token + 语音token → Transformer Decoder           │   │
│  │        → 自回归预测下一个语音token                      │   │
│  └───────────────────────────────────────────────────────┘   │
│                          ↓                                    │
│                       生成的语音                               │
│                                                               │
└──────────────────────────────────────────────────────────────┘

三个阶段：

① Speech Tokenizer 训练（不上文字，只用音频）
    目标：学会把音频压缩成离散token，再还原
    关键：Encoder + FSQ + Decoder
    输出：一个训练好的"音频↔token"转换器

② Text-Speech LM 训练（上文字+音频的配对数据）
    数据：Speech Tokenizer 把训练音频转成的 token 序列
    目标：学会"看到文字→预测对应的语音token"
    关键：Transformer Decoder（自回归）
    输出：一个能从文字生成语音 token 的语言模型

③ 推理（用户使用）
    输入：文字
    流程：Text-Speech LM 生成语音token → FSQ逆量化 → Decoder → 音频
    输出：自然语音
```

### 6.4 和语音识别的对比总结

```
              语音识别 (ASR)              语音生成 (Text-to-Speech)
             ┌────────────────┐         ┌────────────────────┐
输入:         │ 音频 🔊         │         │ 文字 📝              │
             ├────────────────┤         ├────────────────────┤
输出:         │ 文字 📝         │         │ 音频 🔊              │
             ├────────────────┤         ├────────────────────┤
核心组件:     │ Encoder        │         │ Speech Tokenizer    │
             │ + CTC/Decoder  │         │ + Text-Speech LM    │
             │                │         │ + Decoder           │
             ├────────────────┤         ├────────────────────┤
特征方向:     │ 音频→离散token │         │ 离散token→音频       │
             │ (识别方向)     │         │ (生成方向)            │
             ├────────────────┤         ├────────────────────┤
共同技术:     │ 分帧/Fbank/DFT/梅尔滤波（语音识别教材第3-5章）  │
             │ 卷积降采样、Transformer（语音识别教材第7章）      │
             ├────────────────┤         ├────────────────────┤
关键差异:     │ 输出token=文字   │         │ 输出token=音频       │
             │ 用CTC/Decoder解码│        │ 用FSQ量化+LM生成     │
             │                 │         │ +Decoder还原波形     │
             └────────────────┘         └────────────────────┘
```
